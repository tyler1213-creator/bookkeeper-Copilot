# 任务：第一性原理审计 【本轮节点：____】 节点 → 产出 L2 Proposal

> **复用说明**：每审计一个新节点，只替换全文【】内的占位项
> （节点名、节点工作区目录名、本节点的 B 类核心审计对象文档、本节点相关的「已锁事实」、本节点相关的圈外/未落文对象）。
> 其余结构与铁律一律不动。
>
> **本任务的真正目的**：不是去挑 New System 文档的毛病、也不是去优化它，而是**设计出一个好的节点**。
> 「这个节点到底应该怎么运作」的答案只来自两处：① 已落文正式文档锁死的硬契约（A 类）；② 用户的看法与判断。
> New System / Old System 只是**热身参考**，帮你快速熟悉节点在说什么，读完即放下。

---

## 一、权威分层（贯穿全程的第一铁律）

所有材料分三类，真相锚永远在 A 类；B 类不进设计、不进契约面。

- **A 类 · 权威源（真相锚 / 唯一设计约束来源）**
  `BK_Copilot/` 下**已落文邻居**的正式 workflow_node / memory_layer 文档（00/01/02）、模板规则、以及本任务声明的「已锁事实」。
  **节点的输入/输出契约面只能从 A 类重建。**

- **B 类 · 待审计材料（仅热身 + 提问，永不进设计）**
  `new system/` 设计、`old_system_nodedesign/` 设计。合法用途只有两个：
  1. **行话 / 领域热身**——快速看懂这节点大概在干嘛、术语怎么说。
  2. **问题发生器**——它哪怕模型是错的，也可能提醒一个值得问的真问题；**只能提出问题，永不提供答案**。
  **绝不能**当结构模板（不得继承其字段名、枚举、状态机、输入/输出形状），**绝不能**当正确性证明、baseline 或可复用约束来源。

- **铁律**
  1. 凡 Proposal 中出现的任何契约结构，必须能逐条指回**某条 A 类原文**，或指回**用户已拍板的决定**；两者都指不回的，不许写。
  2. B 类与 A 类（或某条「已锁事实」）冲突时，一律以 A 类为准，并按「停止并问用户」停下报告，不得用 B 类模型覆盖。
  3. 「这份材料不权威」这个标签本身不够——必须**主动拒绝沿用它的结构**。详细但错的，不得挤掉抽象但对的。

## 二、执行架构（sub-agent 读档、主线程判断与讨论）

- **只有「读档」外包**：规则文档、A 类邻居正式契约、B 类核心审计对象、（仅当用户指定时）Old System，由 sub-agent 代读。
- **判断 / 交互 / 成文不外包**：推邻居体检由 sub-agent 机械执行，但 A 桶清单需用户确认；吸收用户观点、梳理问题、L1-L4 分类、讨论、写 Proposal，全部主线程（你）做。
- **sub-agent 只蒸馏不预判**：只交 ① 逐字引文 + 出处(文件:章节) ② 结构化索引 ③ **从 A 重建的契约面 + 缺口标注**（只摆 A 原文、只标缺口，不下判断） ④ B 类热身/防污染小抄（用完即弃）；不得产出问题清单或最终判断（问题须在主线程吸收用户观点后、以正式文档模板为标尺才能成立；语义是否对齐属主线程判断）。
- **产物落盘**：sub-agent 把详档写进 `_audit_work/【节点工作区】/`，只回传「≤300 字摘要 + 文件路径」，不把大段原文塞回主线程；主线程对存疑处按路径回读源文件。

## 三、整体流程（务必按序，不可一上来就写文档）

**第 1 步 信息读取与输入**
- 1a 规则文档（Agent A）、1b A 类契约面 + 缺口 + B 类热身（契约面 Agent）：由 sub-agent 代读（见 Phase 0）。
- 1c 用户观点：Agent 读完并汇报摘要、且 A 桶清单经用户确认**之后**，用户会主动给出他对本节点的看法（他认为这节点该怎么转）。**等待并吸收 1c 后，再进入第 2 步。**
- 1d Old System：仅当用户指定具体文档时才派 Agent D 读，否则跳过（按 B 类对待）。

**第 2 步 问题梳理（主线程）**
- 问题清单 = **`01` 里的 △缺口（A 锁不住、要用户判断的开放问题）** + **`02` 里 B 提的候选问题** + **用户 1c 的判断**，以「要按 BK Copilot 正式文档模板写出本节点文档，还差哪些信息没定 / 定义不清 / 不合理」为标尺列出。
- **硬契约面（A 已锁死的）不拿来讨论，直接当约束；缺口才是讨论的主战场。**
- 每个问题严格分类 L1 / L2 / L3 / L4。

**第 3 步 编写 L2 Proposal（主线程）**
- 一切以 BK Copilot 正式文档模板为标准；目标是集齐「足以转成正式文档」的全部信息。
- 把问题**每 5 个一组**发用户逐一讨论；讨论清楚、用户确认后，才把结论写进 Proposal。
- 先讨论完所有 L1、L2 问题再动笔。**提案里每条契约结构，要么指回 A 原文、要么指回用户拍板的决定；B 永远不能当依据。**
- 当 Proposal 内容已足够支撑创建本节点从 L1 到 L2 的正式文档时，本任务结束。

**第 4 步 审核与成文（不在本 Agent 范围）**
- 用户会与其他 Agent 审核 Proposal，再按模板转成正式文档。本 Agent 不创建 / 编辑任何正式文档。

## 四、Phase 0 派发（第 1 步读档外包）

工作区：`_audit_work/【节点工作区】/`。sub-agent 只读 formal 正式文档与指定材料，**不读 L2 提案**（提案是讨论存档；正式文档才是当前权威态）。

### Agent A — 规则蒸馏 → `00_rules.md`

读：`当前任务状态.md`、`AGENTS.md`、`审计目标与原则.md`、`系统上下文地图.md`、`项目背景速读.md`、`缺口地图.md`、模板 `BK_Copilot/node:layer spec template/workflow_node_spec_template_rules.md`。

产出：
1. 必守通用规则（source authority 铁律、可编辑范围、停止/升级、已锁不变量、守 seam、契约面、不把 new/old 当权威）逐条 + 出处。
2. 模板 Stage 1-2 完成清单 + 禁止写法，逐字搬运——这是第 2 步梳理问题的标尺，也是 Proposal 自检表。

### 契约面 Agent（原 Agent B）— 邻居体检 + 从 A 重建契约面 + B 类热身

按以下三步，**A 桶与 B 类全程分开、优先级分明**：

**步骤 1 — 推邻居 + 落文体检（机械执行，结果交用户确认）**
- 从 `系统上下文地图.md` 的链路 + 本节点 B 文档自己声明的「我消费谁 / 产出给谁 / 读写哪些 log」（B 在此**只用来提示邻居是谁，不用来定契约内容**）+ `缺口地图.md`，推出本节点的邻居名单。
- 拿 `系统上下文地图.md` 里「已有正式草案」清单逐个对：
  - 邻居**已落文** → 进 **A 桶**（读其正式 00/01/02）。
  - 邻居**未落文**（如尚未立草案的节点 / 概念文档）→ 进**未落文依赖清单**，挂 open boundary，**绝不许拿 B 填**。
- 别忘了把 `AGENTS.md` / `审计目标与原则.md` 里与本节点相关的**已锁事实**也算进 A 桶。
- **回传 A 桶候选清单 + 未落文依赖清单给用户，先确认再深读。**
- **硬停条件：A 桶任一权威源未能按路径定位（含报告"未找到"）→ 立即停下报告，不得用 B 类或任何非权威材料填补该空缺。**

**步骤 2 — 从 A 单独重建契约面 + 标缺口 → `01_契约面与缺口.md`**
- 逐条接缝（A 桶每个已落文邻居 ↔ 本节点），**只抄 A 的原文**：这条接缝 A 锁死了本节点要提供什么 / 邻居给本节点什么。
- 枚举必须**从 A 的完整输出/输入清单逐项过**（这样 A 锁了、本节点却没接的，会暴露）。
- A 没说 / 邻居未落文的接缝 → 标 **△缺口**（这是要交给用户判断的开放问题，是本步最值钱的产物）。
- **B 类在这份文件里一个字都不出现。**
- 语义是否对齐（"邻居输出的 P 到底满不满足本节点的需要"）属判断，**不在此下结论**，只把两端 A 原文摆清楚，留主线程对齐。

**步骤 3 — 读一遍 B 热身 → `02_热身与防污染.md`（用完即弃）**
- 读：`new system/node_stage_designs/【本节点】__design.md` 及相邻 new system 设计（全部标 `[待审计·不具权威]`）。
- 产出三样，都是辅助、非设计输入：
  1. **行话 / 领域速记**：帮主线程快速进入状态。
  2. **B 提醒的候选问题**：B 触及、值得问用户的点（**只列问题，不给答案**）。
  3. **防污染小抄**：B 的说法与 A 明显分叉处，逐条标"此处别顺着 B 走，以 A 为准"——这是给主线程的护栏，**不是 B 的缺陷待办**。

### Agent D — Old System（仅当用户指定时才派）→ `03_old_system.md`

读：【用户指定的 Old System 关于本节点的文档】。逐字摘相关设计 + 出处（注明：历史材料，不具权威性；按 B 类对待——仅热身 / 提问，不进设计）。

## 五、边界与约束

1. **契约面**：本节点既消费上游、又产出下游。上下游各提供 / 输出什么、各 consumer 是谁，由 `01` 契约面（从 A 重建）+ 主线程推导，写成「上下游依赖 → 本节点输入/输出」映射（Proposal 的一部分）；**契约面只能从 A 类重建**，每个输出写明 consumer，且不与已锁下游抵触。
2. **已锁事实（遵守，非审查对象）**：【列出本节点相关的已锁事实，并注明「具体适用范围以对应正式文档为准」。例：身份判断 stable/unknown 锁在 ER 侧；会计分类归下游；AI 联网 / external lookup 锁在 ER 侧……】。
3. **Scope / seam**：本轮只审本节点侧——读什么、产出什么、谁有权威。涉及独立 memory layer 的存储 / 写入机制属其对象范畴，本轮不重定义，守运行 / 记忆 seam。
4. **圈外 / 未落文对象只挂起引用，不强行闭环**：【列出本节点相关的未落文对象，例：Coordinator、Profile、Governance，以及相邻但尚未单独设计的节点】。这些的契约**绝不许拿 B 填**，一律挂 open boundary。

## 七、停止并问用户

- 出现需真实产品决策（而非措辞）的点。
- **B 类材料假设的输入/输出契约，与某上游/下游正式契约或某条「已锁事实」冲突 → 停下报告，按 A 类权威为准，不得沿用 B 类模型。**
- **任一关键 A 类权威源未能按路径定位 → 停下报告，不得用非权威材料填补。**
- 设计与已锁下游契约冲突，且权威顺序无法解决。
- 任一 output 的 consumer 无法从已锁结论确定。
- 发现正式文档之间、或与已锁不变量矛盾 → 停下报告，不得以 Proposal 覆盖正式文档。
