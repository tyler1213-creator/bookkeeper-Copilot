# 任务模板：审核「{{本轮对象}}」L2 提案的「生成就绪度」（通用·一次一个，审完即停）

> **使用说明**：本模板适用于审计任意节点 / 记忆层的 L2 提案。开审前先跑 §0（多数参数可由你自动发现 + 与用户确认，不需手填）；§1 起为通用审计机制，引用 §0 的参数。
> **第一性原理铁律**：本模板**不写死任何业务级已锁不变量**。凡涉及"当前锁了什么"，一律从权威来源现取（§0.5），**不以本模板的任何快照为准**——这样模板不会随系统演进而腐烂。

---

## §0 开审前：配置与自动发现（先做，确认后再开审）

逐项确定本轮参数；能自动发现的先发现、再交用户确认，不默认合法。

- **0.1 本轮范围**：审一个节点、一组相关对象（圈），还是 `L2_proposals/` 下全部。默认＝用户指定；未指定则列全部交用户圈定。
- **0.2 目标清单（自动发现）**：列出 `bookkeeper-Copilot/L2_proposals/` 下所有提案。唯一正确位置是此目录；若在外侧 `system_audit/L2_proposals/` 发现文件，**本身即记为 finding（放错位置）**，不是正常搜索点。清单缺失 / 重复 / 出现范围外对象 → 先标出问用户，别默认合法。
- **0.3 依赖顺序（自动推断）**：从各提案「依赖」节推断拓扑序，上游先审。与用户确认；无明确依赖则用文件给定序或问用户。
- **0.4 每对象的审计模式（自动判定）**：到 `bookkeeper-Copilot/BK_Copilot/` 查该对象正式草案是否存在。
  - 存在 → 审「改」的就绪度 + 拟改动与现状一致性。
  - 不存在 → 审「按模板从零建」的就绪度；其「拟改:文件:章节」指向**待新建**文件，核对的是"信息够不够建"，而非"与现状是否一致"。
- **0.5 本轮已锁不变量（现取，勿信模板快照）**：从权威来源推断当前已锁不变量并与用户确认。来源优先级：`当前任务状态.md` > 各对象已落 `BK_Copilot/` 正式文档 > 已审通过的 L2 提案。把它们列成"违反即判失败"清单——后续 §4 引用。
- **0.6 圈外 / 外阻对象**：列出本轮不该被触碰、只能挂起引用的对象（从范围 + 依赖推断 + 用户确认）。
- **0.7 跨对象接缝清单**：仅当本轮含多个相关对象时适用。从各提案声明的 input/output 契约面与依赖，自动识别相邻接缝，留待 §8 最终接缝审。

---

## §1 你的角色与这次审什么

你是**审查者 / 检察官，不是实现者**。提案作者已把本轮各对象的 L1-L2 结论写成提案（推理链 + 拟对正式文档的改动）。你一个一个审，每个提案同时回答两个问题：

- **A. 决策是否正确、守纪律**（能不能成立）。
- **B.（本轮核心）信息是否足以驱动正式文档**——即一个实现者只拿「这份提案 + 已锁上游 + 模板」，能否不靠猜就：(1) 改对应的 `BK_Copilot` 现有正式文档；或 (2) 按模板从零建该对象的 L1-L2 文档。并明确列出：哪里信息有误、缺哪些必须补的信息、应对目标文档做哪些具体改动。

你不做设计、不写正式 spec、不自行改文档（除非用户明确要求）。

## §2 最重要的纪律：一次只审一个

按 §0.3 的依赖序，一次只审一个；给出结论后**停下**，等用户说「下一个」再继续。不要一次性读完或审完所有提案。

## §3 读取协议（分层，别一次读完）

- **A. 开局只读判据（小、强制）**：`AGENTS.md`、`审计目标与原则.md`、`当前任务状态.md`、`缺口地图.md`、两份模板 `BK_Copilot/node:layer spec template/*`（重点 L2 / M1-M2 完成清单与「禁止写法」）。
- **B. 每审一个对象，只按需读**：该对象提案 + 它拟改 / 拟建的 `BK_Copilot` 目标文档 + 已审通过的上游提案结论 + 相关那一份主题记录。仅在需追溯结论来历时才读对应的主题存档（如各对象的「相关问题」记录、`Coordinator Question.md` 之类）。
- **C. 审下游时**，把已审通过的上游提案决策当**锁定基线**；下游与上游矛盾 → 判失败。

## §4 不可动摇的前提（违反即判失败）

**两类，分清**：

**(a) 永远适用的结构性纪律（与具体对象无关，恒检）：**
- **契约面**：每层只通过显式、最小、写明的接口对外，下游不得依赖其内部。
- **运行 / 记忆 seam**：运行节点只声明「记什么 + 谁有权威」，记忆层只定「怎么记 / 谁记 / 什么顺序」，两者不得越界或互相矛盾。
- 不把 new / old system 当权威；**未脱离来源重新论证就搬旧设计＝违规**。
- 不越界动 §0.6 的圈外对象，只能挂起引用。
- 本轮范围只到 L2（node Stage 2 / memory M2），**不进 L3**。

**(b) 本轮业务级已锁不变量**：见 §0.5，从权威来源现取并经用户确认；违反即判失败。**以最新来源为准，不以本模板任何文字为准。**

## §5 审查维度

按两组逐项查。

**第一组——正确性 / 纪律（决策能不能成立）：**
- **复活已废概念**：可 grep 查残留；废弃概念清单从 §0.5 不变量推断（凡某不变量"删除/禁止"了某概念，即扫其残留）；grep 时排除被审计材料目录（`new system/`、`old_system_nodedesign/`）与泛化治理用法（如 rule promotion candidate）。
- 在 L2 **冻结字段级 schema**（越权到 L3）；或把 L3/L4/外阻错标成 L2 硬做；或把**真 L2 推托成 park**。
- 违反 seam；该 park 的机制硬做（多 log 写入、exact writer、trigger order、存储形态等）；对外阻项**强行闭环而非挂起**。
- 把 new / old system 当权威，或未脱离来源重新论证就搬旧设计。
- 越界 / 动了圈外对象；没锚定核心产品目标；替代方案排除理由不成立。
- 触发模板「禁止写法」（LLM 自行判断 / 必要时更新 / 高置信度自动沉淀 / 写日志但不说 authority / 生成候选但不说谁消费谁批准 等——以模板现行清单为准）。

**第二组——生成就绪度（够不够写文档，本轮核心）：**
- **契约面完整**：每个声明的 output 都写明 consumer 与下游含义；每个依赖的 input 都写明 source / authority。缺 consumer / 缺 source 即记为生成阻塞。
- **模板必答项覆盖**：对照模板 L2 / M1-M2 完成清单，逐条核对该提案是否已提供答案；列出未过项（清单你自己跑，不要求提案抄清单）。
- **实现者会被迫猜的点**：站在「只拿这份提案 + 锁定上游 + 模板」的实现者视角，列出他写文档时缺的、必须补的信息（语义级，不是 L3 字段级）。区分「这是 L2 该补的」与「这是本就留 L3/L4/外阻、文档里应标 open boundary 的」——后者不算阻塞，但要确认提案已正确标注。
- **「改」模式专项**：逐条核对「拟改:文件:章节」——目标文件 / 章节是否存在？周边现状文字是否与拟改动冲突（含提案自陈的已知偏差）？悬空引用？
- **「建」模式专项**：提案是否提供按模板新建 `00_index / 01 / 02` 所需的全部语义（存在理由、唯一职责、读 / 写 / 权限 / 异常 / trace / 契约面 / open boundaries）。
- **跨提案一致性**：本提案引用的上游契约语义，与已审上游提案是否字面一致（命名、状态枚举的语义、authority 归属）。

## §6 篇幅中立

提案价值在真决策对不对、信息够不够，不在长短。别因短扣分、因长放心。若为凑格式造稻草人替代、或把完成清单抄进提案，一句话提醒「此处格式填充，可精简」，但非阻塞项，别喧宾夺主。

## §7 输出格式（每个提案一份；每条指到 文件:位置 + 问题 + 违反哪条纪律 / 缺哪条信息 + 建议；不替用户拍板）

**决策类裁决：**
- ✅ 决策成立（过清单）　🟡 成立但有缺口　❌ 做错 / 越界（违反前提 / 纪律）
- ⏸️ 分类错误 / 该 park 却硬做 / 该 L2 却推托　❓ 把判断当结论、该用户拍板
- 🔁 与已审上游不一致 / 悬空引用 / 概念残留（附 grep 结果）

**生成就绪类（本轮核心，必给）：**
- 📋 **生成所需补充信息**：实现者据此写文档时会被迫猜 / 必须补的点（逐条，标「L2 该补」或「应标为 open boundary」）。
- 🛠 **对目标文档的具体改动**：该提案落地时应对 `BK_Copilot` 文档做的改动；「改」模式标明与现状的冲突点，「建」模式标明按模板新建哪些文件 / 章节。
- 📌 **待用户决定的问题**。

**末尾两条总评，分开下：**
- 决策是否成立：可 / 有条件（列条件）/ 否。
- 是否足以生成文档：足够 / 有条件（列必补信息）/ 不足。

然后**停下**，等用户说「下一个」。

## §8 触发「最终跨对象接缝审」

**仅当本轮含多个相关对象、且用户说「所有提案都各自过关」时**触发。做一次跨对象审，专查 §0.7 识别接缝的**生成就绪度**。通用接缝类别：
- 上游 output ↔ 下游 input：相邻契约语义是否对得上、字面一致。
- 跨对象共享的状态 / 字段语义（如状态枚举、policy、索引键）是否各处一致。
- authority 归属是否不重叠、不冲突（谁写、谁批、谁只读）。
- merge / split / 生命周期变更等跨对象联动是否有一致落点。

重点：相邻对象能否**各自独立生成而不打架**。

## §9 边界

报告 findings，不替用户决定，不自行改 spec（除非用户明确要求修）；需产品决策的报为**待决**，不硬闭环；不把被审计材料当权威。
