# L3 阶段执行计划

> 从零到完成的完整方案。本计划是本轮讨论收敛出的版本——按"先理解功能 → 再追值的流转 → 最后起名"展开，不是按节点孤立设计字段。

## 总目标

把每个必要节点 + 记忆层的字段 schema 定下来，做到两件事：
- **目标 1（竖着读 = 节点契约）**：每个节点的字段能承载它的全部已冻结功能。
- **目标 2（横着读 = 字段链路）**：每个值从生到死一个名字、有头有尾、含具体值。

最终所有跨节点字段真值集中在**一张唯一权威大表**里，正式 `03 / M3` 文档由它派生。

## 贯穿全程的作业原则

- 一个节点 = 一个子 workflow，内部由若干"块"按顺序拼成。
- 文档冻结的是语义 / 功能，不是字段名；名字到最后一步才起干净的。
- **留名**（跨边界、有人依赖的契约数据 → 字段）vs **草稿**（块内部实现 → L4）。
- 动作六词：`create` / `remake` / `pass` / `read` / `set` / `save`。
- **主链先行，分支后补**；卡在圈外对象的"尾巴"字段——名字先定、形状挂开口，不硬冻。
- 作业台 `L3_schema/` 是工作层；稳定后才回写正式 `03 / M3`，两者不混。

---

## 执行模型（新窗口必读，先读这节再动手）

这套 Phase **不是一个窗口从头跑到尾**，也**不是每个节点都重跑一遍 Phase 0–5**。拆开看：

- **Phase 0 = 一次性全局**，已完成。新窗口**绝不重做**。
- **Phase 1 + Phase 2 = 逐节点循环，是新窗口的标准工作单元。一个窗口只做一个节点**：把这个节点拆解（Phase 1 六步）+ 立刻把它接进大表（Phase 2 那一段）。两者绑在一起做，不是"先拆完所有节点再统一接表"。做完存档、更新进度，窗口结束。
- **Phase 3 / 4 / 5 = 里程碑活动，不在每个节点窗口里做**，由 owner 在合适时机单独发起：
  - **Phase 3（样例验证）**：主链节点都接进大表后先验一次，**不必等所有分支节点**。
  - **Phase 4（回写正式 03/M3）**：某节点链路稳定后做，可逐个也可集中。
  - **Phase 5（完成判定）**：全部收口时的最终 gate。

> 一句话：Phase 0 全局一次 → 沿主链「拆一个节点 + 接进表」循环（每个新窗口一个节点）→ 主链通了验一次 → 稳定了回写 → 全收口判定。

## Phase 0 · 立规矩 + 搭作业台　〔基本完成〕

**目标**：定清楚"L3 怎么做"和"什么算做完"，防止变成新的文档泥潭。

**步骤**：
1. 建作业台目录 `L3_schema/`，与正式节点草案分开（防影子权威）。✅
2. 定动作词表（create / remake / pass / read / set / save）。✅
3. 定"留名 vs 草稿"纪律。✅
4. 写《节点 L3 拆解模板 Prompt》，每个新节点照跑。✅
5. 定 L3 完成标准（见 Phase 5）。✅

**产出**：`L3_schema/` 目录 + 模板 Prompt + 动作词表 + 完成标准。

---

## Phase 1 · 逐节点拆解（沿主链，一个节点一份 schema）

**目标**：每个节点 / 记忆层产出一份 `{节点}_L3_Schema.md`：内部子流程图 + 锁定字段清单 + 各块「吃进 / 吐出 / 留名」。

**顺序**（按交易主链）：
Evidence Intake ✅ → Entity Resolution → Rule Match → Case Judgment → Coordinator → Human Review Node；
共享机制 **Finalization**（落盘地板，是尾巴字段和记忆层落点的关键解锁项，排在写入阶段前后优先做）；
记忆层落点 Transaction Log / Entity Log / Case Log / Rule Log / Alias Log。

**每个节点的步骤**（= 模板 Prompt）：
1. 读档（`00/01/02` + 缺口地图该段；L2 提案按需查）。
2. 大白话讲清节点干嘛、进出什么 → 确认。
3. 拆成内部块、出子流程图 → 确认骨架。
4. 一块一块过 吃进 / 吐出 / 留名 / 草稿 → 逐块确认。
5. 并字段清单（主干 / 旁路）、起干净名 → 逐个拍名。
6. 存档 `{节点}_L3_Schema.md`。

**产出**：每节点一份 L3 schema 文件。

**两点 scope 提醒**：
- 先走每个节点的**主路径**；分支（unknown / pending / 纠错 / 纠正）等主链通了再补。
- **记忆层**不套"六块流水线"——它的 schema 是"记录字段 + 生命周期"，由主链 `save` 进来的东西喂；按这个适配版来。

---

## Phase 2 · 跨节点拼接（那张唯一权威大表）

**目标**：把各节点字段接成一张以"值"为行、"节点"为列的链路大表，作为跨节点字段真值的**唯一权威**。

**步骤**（与 Phase 1 配对、沿主链逐段做，不是等所有节点拆完才开始）：
1. 每拆完一个节点，就把上一节点的输出接到本节点：每个值在本节点是 `pass` / `read` / `set`，还是 `create` 出新值。
2. 同一个值跨节点必须**同名**；发现同值不同名（如 `direction` 的 money_in/out vs inflow/outflow）**当场统一**。
3. 每个值必须有"出生"格和"落点"格；断头线、无主字段当场暴露并修。
4. 大表锁定后，各节点 schema 文件的字段**指向大表**，不再各自定义。

**产出**：`L3_schema/字段链路大表.md`（唯一权威）。

**先主干后分支**：先把 `transaction_id` / `entity_id` / `amount` / `accounting_treatment` 等主干值走通，再织入分支值。

---

## Phase 3 · 具体值跑通验证

**目标**：用真实样例交易，验证 schema 真能从头流到尾、字段够用。

**步骤**：
1. 选一笔典型交易，填真实值（编号、金额、方向、entity 等）。
2. 沿主链一格一格走，看每个节点拿到的形状接不接得上。
3. 抓出三类问题：缺字段、同值换名、不该进 L3 的字段。
4. 至少再跑一条分支（如 unknown 身份 / pending）。

**产出**：`L3_schema/examples/` 下几个端到端样例。

**完成标准**：样例能不卡壳地从 Evidence Intake 流到落盘——被开口挡住的尾巴除外，且开口必须写清依赖谁。

---

## Phase 4 · 回写正式 03 / M3

**目标**：把作业台里稳定的结果，落进正式 spec。

**步骤**：
1. 某节点字段经大表 + 样例验证稳定后，写其正式 `03_data_contract.md`（节点）或 M3（记忆层）。
2. 正式文档的字段定义**指向大表**，不重新发明。
3. 更新该节点 `00_index` 的 Stage 3 勾选 + 缺口地图对应 L3 项收口。

**产出**：各节点 / 记忆层的正式 `03 / M3` data contract。

**纪律**：作业台 ≠ 正式权威；只有回写后才算正式冻结。

---

## Phase 5 · 完成判定（L3 收口 gate）

**L3 算完成，必须全部满足**：
1. 每个必要节点 + 记忆层都有 L3 schema，且字段覆盖其已冻结功能。
2. 跨节点大表存在，每个共享值从生到死一个名字、有出生有落点。
3. 主链至少一个具体值样例端到端跑通。
4. 同值不同名全部合并 / 解释；`direction` 等枚举口径统一。
5. 卡在圈外对象（JE Generation / Finalization L3 / Intervention Log 等）的尾巴字段，全部以"开口 + 依赖谁"显式挂着，**没有假冻结**。
6. 稳定结果已回写正式 `03 / M3`；未决项回缺口地图，不混在 schema 里假装已定。
