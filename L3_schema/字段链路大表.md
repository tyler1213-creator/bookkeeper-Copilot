# 字段链路大表（跨节点字段真值唯一权威）

> **作用**：横着读——每个值从生到死一个名字、有出生有落点、含动作。这是跨节点字段真值的**唯一权威**；各节点 `{节点}_L3_Schema.md` 的字段指向本表，正式 `03 / M3` 由它派生。
> **HTML 审核视图（非权威）**：本表有一个供人审核的 HTML 派生视图 `字段链路审核视图.html`（同目录）。它**非权威**——Schema 设计一律以本 `.md` 为准；该视图仅在 owner 单独明确指示时、依据本表填值更新，**不得让视图领先于本表，也不得把它当设计权威或读取来源**（见 Decisions D12）。
> **状态**：draft，沿主链逐段建。已建：① **Evidence Intake → Entity Resolution**；② **Entity Resolution → Rule Match**（本轮接入）。
> **动作词**：`create`（凭空新建）/ `remake`（从上游算出新名新值）/ `pass`（原样透传）/ `read`（读用不改）/ `set`（同名敲定）/ `save`（落库）/ `—`（未出现）。
> **结构**：按段分节，每节内「值一行，含动作 + 出生 → 落点」。一、EI → ER 段；二、ER → Rule Match 段；后续节点（Case Judgment …）通了再加段。

## 二、ER → Rule Match 段

> 只有 ER 输出 `identity_state=stable` 的交易进入 Rule Match；`unknown` 直接走 Case Judgment、不过 Rule Match。
> 另：被 EntityLog `force_pending` 命中的交易，在 ER→RM 之间的 **eligibility router** 已改道 Pending，不进 Rule Match（Decisions D8）。Rule Match 是 runtime-only、自身不写库；命中交易由后续 finalization 落 Transaction Log。

### 从上游流入、被 Rule Match 处理的值

| 值 | Entity Resolution | Rule Match | 出生 → 落点 |
| --- | --- | --- | --- |
| `entity_id` | create（新建）/ read（复用）——身份输出 | read（查 RuleLog 的键）+ 命中时随 rule-hit handoff 输出（= StableEntity ref） | ER 出生 → RM 用作键 + 携带下游 |
| `direction` / `amount` / `transaction_date` / `currency` | read（随交易透传） | read（匹配维度）+ 命中时随 handoff 携带 | EI 出生 → RM 匹配 + 携带 JE Generation |
| `transaction_id` | pass | pass（全程锚） | EI 出生 → 透传全链 |
| `raw_description` | read / pass | pass（只随 trace，不作匹配条件） | EI 出生 → 透传 → finalization |
| `evidence_refs` | read（投影出 `identity_evidence_refs`） | pass（trace / 审计，不匹配） | EI 出生 → RM 透传 |
| `identity_reason` / `identity_evidence_refs` | create（ER 产） | pass（RM 不读不处理，透传） | ER 出生 → 穿过 RM → finalization 落 Transaction Log |

### 在 Rule Match 诞生 / set 的值

| 值 | Rule Match | 出生 → 落点 |
| --- | --- | --- |
| `rule_match_outcome` | create（命中数闸门：0→rule_miss / 1→rule_hit / ≥2→invalid_handoff·blocked） | RM 出生 → 路由（hit→JE Generation、miss→Case Judgment、blocked→review/gov） |
| `rule_id` | read 自 RuleLog → 命中时输出（命中规则 id，不钉版本） | RuleLog 来源 → RM 输出 → JE Generation + finalization（审计） |

| `classified_by` = `rule_match` | set（表「本笔分类由谁 / 什么做出」= 规则匹配；与 `confirmed_by`〔最终结果确认 / 审计〕分离，见 D11；仅 rule_hit） | RM set → 随交易落 Case Log / Transaction Log |

## 三、口径与断头检查

- **同值同名**：`raw_description` 在 EI / ER 同名透传，不漂移；运行期表面文字只此一名，`alias` 仅为其写入两个 log 时的存储记录名（不是运行期改名）。
- **`identity_evidence_refs` 为何不直接用 `evidence_refs`**：它是身份子集 + 并入 C 的外部来源 ref（EI 的 `evidence_refs` 无外部来源），属 `remake`，故单列。
- **每个值有出生有落点**：上表无断头线、无无主字段。
- **`direction` 枚举口径**：EI 标注「内部取值口径待与 Rule Match / JE 统一」；Rule Match 现作匹配维度 **read** 它，但跨节点枚举统一仍留 JE 段（RM 不在此冻 `direction` enum）。
- **`evidence_refs` vs `identity_evidence_refs`（RM 段确认）**：二者不同值——前者 EI「交易→源材料」指针、后者 ER 身份投影；在 Rule Match 内**均 pass、都不作匹配条件**，不混。
- **RM 输出不止结局信号**：`rule_hit` 时连同 `rule_id` + `approved_accounting_treatment` + `entity_id` + 客观事实一并吐给 JE Generation（见 ER → Rule Match 段）。

## 四、本段挂起的开口（依赖谁，不在本轮硬冻）

- `alias_record` 的 exact field schema、normalization / equivalence / 冲突修正 / supersession → **Alias Log L3 / L4**。
- `created_by` 的 exact enum、`entity_record` / 最小创建 provenance schema → **Entity Log L3（M3）**。
- `identity_evidence_refs` 外部来源 exact 字段（URL / source title / retrieved_at / snippet）→ **Stage 3**。
- ER 新建实体的同步 finalization 实际写入执行者 / 调用方式 / 两 log 写入顺序 / 幂等 → **L4 / seam**。
- `transaction_id` 等审计材料按 entity 索引检索 → **L4 / seam**（不复制进 Entity Log，见 Decisions D5）。
- alias 查询键确认为 EI 的 `raw_description`（非 `counterparty_signals`）——本段已锁。

**ER → Rule Match 段开口**：
- 上游 **eligibility router**（消费 `force_pending`、改道 force_pending 交易、组装 RM handoff）→ **L4 / seam**；`force_pending` exact enum → EntityLog / RuleLog / Governance 联合 L3。
- `rule_id` exact 格式 → RuleLog M3；`approved_accounting_treatment` exact shape → RuleLog / JE Generator L3；`rule_match_outcome` 枚举 + rule-hit handoff schema → Stage 3；`classified_by` enum（含 `rule_match`，表分类做出方）→ Case Log M3 / Transaction Log L3；`confirmed_by`（最终结果确认 / 审计）→ Transaction Log L3（见 D11）。
- 历史 rule 版本审计还原 = `rule_id` + 交易时间 + Governance Log 回放（要求 Governance Log 记够可重建内容）→ Governance Log L2（见 Decisions D9）。
- `evidence_condition` / `transaction_type` / `objective_tags`：本轮判定 Rule Match 不需要，已删，**不挂开口**（owner 2026-06-26）。
