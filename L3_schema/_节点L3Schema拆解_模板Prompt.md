# 节点 L3 Schema 拆解 · 模板 Prompt

> **用法**：把下面所有 `{{节点名}}` 换成要做的节点（如 Entity Resolution），整段发给 Agent。
> **范例**：已完成的 `L3_schema/Evidence_Intake_L3_Schema.md` 就是标准产出，格式和颗粒度照它来。

---

## 你的任务

为**单个** workflow 节点 `{{节点名}}` 设计 L3 字段 schema，最终产出一个文件 `L3_schema/{{节点名}}_L3_Schema.md`，含：① 节点内部子流程图；② 锁定字段清单；③ 各块「吃进 / 吐出 / 留名」。

整个系统是一条大 workflow，值在节点之间流转；每个节点是一个子 workflow，内部又由若干"块"（子功能）按顺序拼成。这一轮只拆 `{{节点名}}` 这一个节点。

## 底层逻辑（必须遵守）

1. **文档冻结的是"语义 / 功能"，不是"字段名"。** 不要照抄文档里的字段名（尤其又长又拗口的驼峰名）。你要先**理解节点在干什么**、用大白话讲出来，名字到最后一步才由 owner 起干净的。

2. **字段是一根"线"，两头插在不同的块 / 节点上。** 先追"值怎么流"，端口（字段）自然落出来。**不准孤立地先设计某一块的字段**——那会造成命名漂移和返工。

3. **同一个值从生到死只能有一个名字。** 没被加工就一路同名透传；名字要变，必须说清是引用、投影、派生还是新值。

4. **区分"留名"和"草稿"**：
   - **留名** = 跨边界、有人依赖的契约数据 → 升级成字段。
   - **草稿** = 块内部实现细节（算法、解析、哈希怎么算、内部缓冲、执行次序）→ 归 L4，不立字段。文档里凡 park 到 Stage 4 / L4 的，都是草稿。

5. **动作只有六个词**（描述一个值在某块 / 节点里发生了什么）：
   - `create` 凭空产生新值（无上游来源，如 ID）
   - `remake` 从上游值算出一个新名字的新值（如从原始材料归一出金额）
   - `pass` 原样透传，名字内容都不变
   - `read` 读它来用，不改它
   - `set` 同名不变，内容在这里被敲定（如 unknown → 确定）
   - `save` 写进永久存储 ／ `—` 未出现

6. **两个目标同时满足**：竖着读（一个块/节点一列）= 它的输入输出要覆盖其已冻结功能的全部（目标 1）；横着读（一个值一行）= 它从生到死、一个名字到底（目标 2）。

## 行为纪律

- **不准迎合。** 给真实判断，不为顺从而附和，也不为找茬而找茬。
- **一切以文档为准，逐条引用出处（文件:行号），不许凭印象、不许编。** 我说的话若与文档冲突，先去文档核、把原文摆出来，再告诉我哪一半对、哪一半不对。
- **大白话**，少用生僻词 / 英文术语，非用不可先解释。
- **一次只走一步，每步先给我看、我点头才往下**，不要一次性把整个节点全扒出来。
- **我是功能增删的把关人**：拆解中若发现可能要加 / 删功能，标出来给我定，不自作主张。
- **尊重权威边界**：默认只读现行正式草案 `00/01/02` + 缺口地图对应段（结论以这里为准）；**L2 提案不默认通读，按需查**（见 Step 0）；圈外 / 未落文对象只挂"开口"、不替它们冻结；未经我同意不读作废的 new/old system。

## 执行步骤（逐步、每步等我确认）

**Step 0 · 读档**
默认读 `{{节点名}}` 的正式草案 `00 / 01 / 02` + `缺口地图.md` 里该节点段——这是默认权威，不从记忆重建。
`L2_proposals/` 提案**不默认通读**（避免无谓上下文负担），仅在这两种情况按需查对应段：① 正式文档明确指向它（如"见 L2 提案 X.X"）；② 你卡在某个 L3 取舍歧义、需要"为什么这么定 / 排除了哪些替代"的依据时。

**Step 1 · 大白话理解节点**
用大白话讲清楚（按意思，不照抄字段名）：这个节点一句话干嘛的、进去什么、做哪几件事、出来什么。让我不用自己再读一遍就能点头。→ 等我确认。

**Step 2 · 拆成内部子流程（出图）**
把节点拆成内部的"块"（子功能），按处理顺序排出来，画成一张流程图（"几个块 + 闸门 + 旁路分流"那种；用 Mermaid，纯文本可渲染）。说清哪些先后顺序是**冻结**的、哪些精确次序属 Stage 4 没冻结。→ 等我确认骨架（块够不够、顺序对不对、旁路去向对不对）。

**Step 3 · 一块一块拆**
按顺序每块只说四样：**吃进**什么、**吐出**什么、哪些**留名**（升字段，标 create/remake…）、哪些是**草稿**（归 L4）。一块一块给我看，点头再下一块。

**Step 4 · 并清单 + 起名**
把所有"留名"的并成清单，分**主干字段**（跟着干净数据往下游走）和**旁路信号字段**（分流到别处）。给每个字段起干净的大白话名（snake_case；信号带 `_signal` 后缀；按业务含义命名、不按哪个块产的命名；能少则少）。**名字逐个让我拍。**→ 确认后清单锁死。

**Step 5 · 存档**
写进 `L3_schema/{{节点名}}_L3_Schema.md`，排版清爽可读，结构：
1. 顶部简短说明（状态 = draft / 内容 / 口径 / 来源）
2. 子流程图（Mermaid）＋ 冻结的顺序约束 ＋ 贯穿铁律
3. 锁定字段清单（主干 / 旁路两张表，含 含义、出生块、动作）
4. 各块「吃进 / 吐出 / 留名」——只留结果，不写额外解释、不写草稿

---

> **后续跨节点**：等多个节点都拆完，再把每个节点的字段接成一张跨节点大表，看每个值在下一个节点里是 `pass` / `read` / `set` 还是催生 `create` 新值。那张表是字段真值的唯一权威，节点的 `03_data_contract` 由它派生。
