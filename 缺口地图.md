### 交易拆分（Split） → **【DEFERRED · 搁置】**

**场景**：一笔银行交易（Bank Statement 上的单笔）实际由多笔真实交易组成（如两张发票加总 = 银行交易金额），被判定需拆成多笔子交易分别处理。属小众 edge case，早期搁置，不在 Case Judgment Stage 1-2 正式草案展开。下列为搁置前已讨论出的实质，留档备查。

- **已确认边界**：拆分的**检测**与**执行**都不属于 Case Judgment。CJ 无权创建子交易 / 改动交易集合（这是 mutate 交易集合的动作，超出 CJ「对单笔交易做受控分类」的职责）。
- **检测为何必须归上游**：CJ 结构上兜不住复合交易——若两组成部分恰好同科目同税务，CJ 会按单笔高置信度放行而**不自察其复合性**；故「自动发现一笔实为多笔」若要做，必须放在 CJ 上游，倾向 Evidence Intake / Preprocessing（其本就核对证据 / 发票 ↔ 银行流水），不归 CJ。
- **执行归 Coordinator（沿旧系统，历史材料·不具权威性）**：旧系统由 accountant 经 Coordinator 主动提出，`split_transaction.py` 执行，硬约束「各子交易金额之和 = 原始金额」；预处理与置信度分类器都不检测、不执行。
- **拆分后路由（旧系统三支，待 Coordinator 审计时定夺）**：① 会计师已给每条明确分类 → 直接送 JE 生成、不重走；② 只给金额 + 模糊描述 → 生成选项追问；③ 只给金额未说用途 → 子交易作普通新交易从管道上游重入。第 ③ 支的「重走」是未分类交易的正确路径，非形式主义；「拆分后能定分类即直接送 JE」在旧系统第 ① 支已存在。
- **CJ 侧唯一关系**：拆分产生的子交易以普通交易身份（各自 transaction_id + accountant 补充说明作强证据）重入，途经 CJ 时按普通交易判断，无拆分专用逻辑。未拆分的复合交易到达 CJ 时，若 (COA, HST) 不唯一则输出 Pending（reason 可如实说明疑似复合），CJ 不自行执行 / 构造拆分。
- **与 CJ mixed-use 边界的区分**：mixed-use 处理「单笔交易小票内混公司 / 个人品类」是否逐项拆解（默认整笔公司支出、不主动稽查）；本条处理「一笔交易其实是多笔」。两者不同问题。
- **解冻条件**：待 Coordinator 正式审计，或后续产品决定是否引入系统侧自动拆分检测时再处理。

### 无效 / 损坏输入处理 + 错误纠错机制 → **【DEFERRED · 未设计】**

来源：`BK_Copilot/workflow_nodes/case_judgment_node/02_logic_and_boundaries.md` §2 / §12。CJ 定位为系统兜底桥梁，恒输出 High Confidence 或 Pending、不在 Stage 1-2 设计新的拒收路径。以下在 MVP 阶段不设计，待实战检验、发现符合条件的真实场景后再定：

- CJ 对结构性无效 / 损坏输入（如缺 transaction_id、缺 ER 身份结果、缺 authority context）是否拒绝、如何拒绝。
- 被 CJ 与上游节点均无解的交易的最终归处。
- 谁发出纠错 candidate signal、纠错流程如何编排。

### CJ → JE Generation 之间的 COA 校验 → **【L2 行为已定 · 细节 L4】**

来源：`BK_Copilot/workflow_nodes/case_judgment_node/02_logic_and_boundaries.md` §6。已定（含 L2 行为）：High Confidence 进 JE 前过一段确定性代码校验，将 CJ 输出的 COA 与客户 COA 全集匹配——命中放行；**未命中则反馈回 CJ 重判（告知不在 COA 内），仍无法落到合法 COA 则转 Pending**，保证不产生无归处孤儿。校验落点 CJ 末尾独立检查器 vs 并入 JE Generation Node 两可、本 L2 不强制。未细化（L4）：重判次数上限、反馈 payload 形态、纠错 UX 的 exact contract。
