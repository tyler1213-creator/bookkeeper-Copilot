## **Finalization 写入机制**

本 section 是 Finalization（统一 Finalization 写入机制，第三类跨层共享机制）未冻结 L3 / L4 / L2·外阻 缺口的唯一权威池。已于 2026-06-23 审计确认并落入 `BK_Copilot/shared_mechanisms/finalization/` 正式 draft 的 L1 / L2 设计结论不在本文回写；正式边界以该目录为准。审计工作区见 `_audit_work/finalization/`。

### L3（联合 / schema）

- **动作集 exact 定义 + 各动作 payload exact field schema + 动作类型 enum** → **【L3】**（L2 已锁=恰为 `create` / `update` / `append` / `supersede` 四原子动作；是否需 link 已否决=直接填目标 log ID 字段；merge / split 已否决设为地板动作=由四原子动作组合的调用方侧动作批）
- **凭证 exact 形态 + 生成与传递机制** → **【L3】**（L2 已锁=来源仅会计师、挂在整批 payload 上；未定=布尔开关 vs 带签字 ref / 范围、防重放、与"会计师签字"的对应捕获方式）
- **地板回执 exact schema** → **【L3】**（L2 已锁=ID 由调用方铸造，故地板回执**不返还 ID**；回执 = 写入成功 / 失败 + 失败时的结构化原因（哪个动作 / 哪条校验失败，供调用方善后））
- **ID 铸造 scheme + 各动作产 ID 规则** → **【L3】**（L2 已锁=ID 由**调用方**铸造、地板不认身份只写入；L3 定每动作每 log 的 exact ID scheme：`create` / `append`（产生新记录）产生新 ID，`update` / `supersede`（改现有记录）不产生新 ID、只引用现有 `record_ref`；唯一性须由 ID scheme 自身保证（地板不协调），可兼作幂等键）

### L4（实现 / 机制）

- **多 log finalization 写入顺序、原子 / 回滚、幂等键** → **【L4/seam】**（Evidence / Transaction / Case / Rule / JE 顺序；与各对象 section 既有"多 log finalization"缺口同源）
- **"共享库 + 单一事务边界"的 exact 实现** → **【L4】**（L2 已锁=否决"单一全局写手 agent"；实现机制未定）
- **写成功的同步可见性（read-after-write）保证机制** → **【L4】**（L2 已锁=返回成功后记录立即可读；具体保证机制留 L4）
- **失败回滚的 exact 机制** → **【L4】**（L2 已锁=地板回滚 + 如实上报、善后归调用方；重试 / 回滚 / 部分失败呈现等机制留 L4）

### L2·外阻（圈外依赖，挂 open boundary，绝不拿 B 填）

- **Governance Log** 数据层（扩张型变更审计账本，**待建**）+ 其"记录哪些变动"的判定标准 → **【L2·外阻】**（Finalization 侧已锁=扩张型落盘时耦合触发确定性审计写入、由 Human Review Node 扩张改动触发；"何种变动才入账"即触发轴 vs 凭证轴是否一致，归 Governance Log 设计窗口，含 restrictive auto-downgrade 等免凭证变动是否也入账）
- **审核 inbox** 数据层（候选投递目标，确定性发现 job 免凭证写入、Human Review pull，**待建**；草案 `独立question文档/Review_Inbox_question.md`）→ **【L2·外阻】**
- **Intervention Log** 正式 spec（Human Review correction 的 Intervention 记录 + ID 写入目标）→ **【L2·外阻】**
- **merge / split 标准化动作批模板 + 跨 log 迁移完整性** → **【L2·外阻】**（地板只保证整批原子执行 + 各动作校验；"批是否完整不漏项"的语义完整性归调用方 / 治理侧；跨 log 迁移机制归 Entity Log merge/split 契约 + Governance，见 Entity Log / Rule Log section）
- **Finalization 正式 spec 归属目录命名确认** → **【L2·外阻 / 模板】**（第三类跨层共享机制，非 node / 非 memory layer；本轮已按用户指令用 node 模板 00/01/02 骨架 + 四处改写落入 `BK_Copilot/shared_mechanisms/finalization/`，Stage 3 前需确认该目录命名是否继续作为正式归属）

### 交易拆分（Split） → **【DEFERRED · 搁置】**

**场景**：一笔银行交易（Bank Statement 上的单笔）实际由多笔真实交易组成（如两张发票加总 = 银行交易金额），被判定需拆成多笔子交易分别处理。属小众 edge case，早期搁置，不在 Case Judgment Stage 1-2 正式草案展开。下列为搁置前已讨论出的实质，留档备查。

- **已确认边界**：拆分的**检测**与**执行**都不属于 Case Judgment。CJ 无权创建子交易 / 改动交易集合（这是 mutate 交易集合的动作，超出 CJ「对单笔交易做受控分类」的职责）。
- **检测为何必须归上游**：CJ 结构上兜不住复合交易——若两组成部分恰好同科目同税务，CJ 会按单笔高置信度放行而**不自察其复合性**；故「自动发现一笔实为多笔」若要做，必须放在 CJ 上游，倾向 Evidence Intake / Preprocessing（其本就核对证据 / 发票 ↔ 银行流水），不归 CJ。
- **执行归 Coordinator（沿旧系统，历史材料·不具权威性）**：旧系统由 accountant 经 Coordinator 主动提出，`split_transaction.py` 执行，硬约束「各子交易金额之和 = 原始金额」；预处理与置信度分类器都不检测、不执行。
- **拆分后路由（旧系统三支，待 Coordinator 审计时定夺）**：① 会计师已给每条明确分类 → 直接送 JE 生成、不重走；② 只给金额 + 模糊描述 → 生成选项追问；③ 只给金额未说用途 → 子交易作普通新交易从管道上游重入。第 ③ 支的「重走」是未分类交易的正确路径，非形式主义；「拆分后能定分类即直接送 JE」在旧系统第 ① 支已存在。
- **CJ 侧唯一关系**：拆分产生的子交易以普通交易身份（各自 transaction_id + accountant 补充说明作强证据）重入，途经 CJ 时按普通交易判断，无拆分专用逻辑。未拆分的复合交易到达 CJ 时，若 (COA, HST) 不唯一则输出 Pending（reason 可如实说明疑似复合），CJ 不自行执行 / 构造拆分。
- **与 CJ mixed-use 边界的区分**：mixed-use 处理「单笔交易小票内混公司 / 个人品类」是否逐项拆解（默认整笔公司支出、不主动稽查）；本条处理「一笔交易其实是多笔」。两者不同问题。
- **解冻条件**：待 Coordinator 正式审计，或后续产品决定是否引入系统侧自动拆分检测时再处理。

### 无效 / 损坏输入处理 + 错误纠错机制 → **【DEFERRED · 未设计】**

来源：`BK_Copilot/workflow_nodes/case_judgment_node/02_logic_and_boundaries.md` §2 / §12。CJ 定位为系统兜底桥梁，恒输出 High Confidence 或 Pending、不在 Stage 1-2 设计新的拒收路径。以下在 MVP 阶段不设计，待实战检验、发现符合条件的真实场景后再定：

### CJ → JE Generation 之间的 COA 校验 → **【L2 行为已定 · 细节 L4】**

来源：`BK_Copilot/workflow_nodes/case_judgment_node/02_logic_and_boundaries.md` §6。已定（含 L2 行为）：High Confidence 进 JE 前过一段确定性代码校验，将 CJ 输出的 COA 与客户 COA 全集匹配——命中放行；**未命中则反馈回 CJ 重判（告知不在 COA 内），仍无法落到合法 COA 则转 Pending**，保证不产生无归处孤儿。校验落点 CJ 末尾独立检查器 vs 并入 JE Generation Node 两可、本 L2 不强制。未细化（L4）：重判次数上限、反馈 payload 形态、纠错 UX 的 exact contract。
