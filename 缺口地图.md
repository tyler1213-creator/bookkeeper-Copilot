# 缺口地图

**标签**：**【L2】** 本轮收口 · **【L3】** 联合 L3 · **【L4/seam】** 记录层机制 · **【L2·外阻】** 圈外依赖

## **Entity Resolution**

- **stable 阈值** → **【L2】已收口**（见 `L2_proposals/Entity Resolution__L2提案.md`）
- **输出状态两态化 stable / unknown、unknown reason、stable reason、external evidence、exact Alias match、new_stable_entity 及时 publication** → **【L2】已收口**（见 `L2_proposals/Entity Resolution__L2提案.md`）
- **stable entity 确认后的 Entity Log publication 语义** → **【L2】已收口**（实际写入执行者 / 调用方式 / 顺序 → **【L4/seam】**）
- **entity_resolution_output 字段 schema** → **【L3】**
- **new_stable_entity provenance 字段 schema** → **【L3】**
- **knowledge_summary 冲突修复流程** → **【L2·外阻】**（依赖 Knowledge Summary / Governance）

## **Entity Log**

- **identity authority 主档案定位、Entity / Alias authority 与 projection 边界、五类保存集合、保存 / 不保存分流** → **【L2】已收口**（见 `L2_proposals/Entity Log__L2提案.md`）
- **stable entity 创建入口** → **【L2】已收口**（仅 ER `new_stable_entity` 与 accountant explicit identity confirmation；执行机制 → **【L4/seam】**）
- **stable entity lifecycle 最小状态** → **【L2】已收口**（active / merged / archived；exact enum / validation → **【L3】**）
- **automation_policy 的 entity 级控制语义** → **【L2】已收口**（当前控制状态归 Entity Log；存储 / Governance projection 形态与自动降级 exact contract → **【L4/seam】**）
- **merge / split 后 Entity Log 侧身份状态与重定向语义** → **【L2】已收口**（Alias / Case / Rule 跨 log 重判与执行机制 → **【L4/seam】**）
- **person 类型 stable entity 边界** → **【L2】已收口**（是否入 Entity Log 取决于是否经 ER / accountant confirmation 成为 stable entity；Profile ref / projection 形态 → **【L3 / L4】**）
- **entity_record / provenance 字段 schema** → **【L3】**
- **五类保存信息各自的 field schema、enum、refs 形态** → **【L3】**
- **alias_record / Alias 库技术形态、Entity Log 与 Alias Log 物理存储 / projection 边界** → **【L4】**
- **automation_policy 自动降级 mutation contract** → **【L4/seam】**

## **Alias Log**

- **stable entity 确认后 raw transaction surface text 的 Alias Log 写入资格** → **【L2】已收口**（见 `L2_proposals/Alias Log__L2提案.md`；执行机制 → **【L4/seam】**）
- **exact raw description match 直接支持 ER stable identity reuse** → **【L2】已收口**
- **non-exact / normalization / equivalence / vector / semantic matching 是否可直接支撑 stable identity** → **【L3 / L4】**（本轮 L2 已暂定只有 exact match 可直接支撑 stable）
- **多匹配边界** → **【L2】已收口**（Alias Log 不选 winner，交给 ER 判断 stable 或 unknown + reason）
- **merge / split 后 Alias 归属语义** → **【L2】已收口**（merge 跟随；split 治理重判；机制 **【L4/seam】**）
- **Alias 对下游的支持范围** → **【L2】已收口**（唯一直接 reader 是 ER，其他节点只能通过 ER 输出的 stable entity 间接受益）
- **alias_record 字段 schema** → **【L3】**
- **Alias 库技术形态、与 Entity Log 存储边界** → **【L4】**
- **写入责任 / 审批 / 统一 write 机制** → **【L4/seam】**
- **Alias mutation 与 Governance audit / correction / finalization 边界** → **【L4/seam】**

## **Case Log**

- **定位：entity-indexed reusable-precedent layer，非 Transaction Log 的 entity 副本** → **【L2】已收口**（见 `L2_proposals/Case Log__L2提案.md`）
- **write eligibility：所有 stable-linked 且已 finalized 的交易具备写入资格；unknown / pending 不写入** → **【L2】已收口**（见 `L2_proposals/Case Log__L2提案.md`）
- **Case Log 与 Transaction Log 边界：learning view vs audit source of truth** → **【L2】已收口**（见 `L2_proposals/Case Log__L2提案.md`）
- **真值单位：逐笔 case；模式聚合为派生读取视图** → **【L2】已收口**（读取机制 → **【L4/seam】**）
- **`use_level` 复用权限与最小 supersession 复用安全** → **【L2】已收口**（血缘链接 / 执行机制 → **【L4/seam】**）
- **case-derived 信号只提供治理依据，不直接 mutation，不作为 Case Log 存储字段** → **【L2】已收口**（exact governance contract → **【L2·外阻】**）
- **merge / split / archive 后记录归属由治理裁定，Case Log 服从不裁判** → **【L2】已收口**（执行机制 → **【L4/seam】**）
- **保存信息清单（14 项预设字段，字段未冻结）** → **【L2】已收口**（exact schema → **【L3】**）
- **record schema / field / enum / validation** → **【L3】**
- **`use_level` / `confirm_by` exact enum** → **【L3】**
- **共享 accounting outcome（COA / HST / split / allocation）的全局字段结构** → **【L3】**
- **exact writer / Transaction Log trigger order / 多 log 统一写入** → **【L4/seam】**
- **Case Judgment 读取时的 per-entity 固定化聚合（rollup / 轻量展示层 / 检索工具或 subagent）** → **【L4/seam】**（读取层聚合机制；聚合内容全部由逐笔 case 派生，不作为 Case Log 存储字段）
- **supersession 的血缘链接字段与 corrected / reversed / split 后的重判执行机制** → **【L4/seam】**
- **merge / split / archive 后旧 case 重新归属的治理执行机制** → **【L4/seam】**
- **case-derived rule / automation / entity risk candidate governance contract** → **【L2·外阻】**
- **entity / 类别级通则批注归属** → **【L2·外阻】**（依赖 Knowledge Summary / Governance）
- **与 Knowledge Summary 冲突修复** → **【L4/seam】**

## **Rule Match**

- **核心定位、trigger / routing、EntityLog / RuleLog 分工、entity-level / pattern-level scope、non-LLM 执行语义、rule hit / miss / invalid handoff、JE Generator 边界、runtime-only / finalization 写入边界、rule-hit handoff** → **【L2】已收口**（见 `L2_proposals/Rule Match__L2提案.md`）
- **Rule 资格的内容判据**（全函数：scope 可客观判定 + 历史 treatment 唯一无分叉 + 输出 judgment-free 完整 + 事前已批准；次数 / 跨月 / 频率只是证据非门槛；repeated cases / 相似描述 / LLM reasoning / CaseLog 趋势 / 模糊语言只能进 candidate）→ **【L2】已收口**
- **同一 entity 下 rule 集合结构**（允许多条 active rule，互不重叠是写入时不变量；entity-level 与区分性 pattern-level 同 scope 互斥；"默认+例外"建模为互斥 pattern + catch-all；0/1/≥2 命中 = miss/hit/fail-closed blocked，不设 priority）→ **【L2】已收口**
- **RuleMatch 输入 handoff 契约面**（三桶语义：身份基础 entity_id / 放行基础 automation_policy 投影 / 客观交易事实；最终分类 = 命中 rule 的 approved treatment + 桶 C 交易事实，桶 C 必要不充分）→ **【L2】已收口**（exact 字段 schema → **【L3】**；assembler 谁组装 → **【L4/seam】**）
- **RuleLog 查询定位**：全系统以 entity_id 单坐标，RuleLog 按 entity_id 索引，RuleMatch 凭 entity_id 现查候选 active rule 集再用桶 C 比对；EntityLog 不存 rule_id 成员列表（成员关系单一归 RuleLog，避免双真值 / 漂移）→ **【L2】已收口（同时是对 RuleLog 设计的前置约束）**；entity_id 索引的存储 / 检索机制 → **【L4/seam】**
- **automation_policy → RuleMatch 放行**：由 EntityLog automation_policy 承载（不单设 supports_rule_match 字段），"放行 RuleMatch"是其取值语义 → **【L2】已收口**；"哪种取值 = 允许 rule-based automation"的 exact 取值集合 → **【L3】**（RuleLog / EntityLog 联合，需对齐 Governance）
- **RuleMatchNode exact input / output fields、rule-hit handoff schema、rule_hit / rule_miss / blocked enum、accounting treatment（judgment-free 完整）schema、catch-all pattern 语法糖、资格阈值具体数字** → **【L3】**
- **RuleMatch matching algorithm、condition validation、amount range validation、rule 集合 overlap-validation、ER handoff / workflow router / RuleLog reader 调用顺序、多 log finalization、RuleLog 与 EntityLog 同步写入机制、rule hit count / rule health 统计维护机制** → **【L4/seam】**
- **Rule promotion eligibility、approval workflow、candidate queue、rule lifecycle、rule modification / downgrade / deletion、rule staleness / 前瞻有效性、case-derived rule promotion evidence governance contract** → **【L2·外阻】**（依赖 RuleLog / Governance）
- **下一步**：RuleMatch 节点侧 L2 已实质完成；闭环须先开 **RuleLog L1-L2**（资格判据的最终归属、entity_id 索引语义、automation_policy 取值语义、rule lifecycle 都在那里落地）。BK_Copilot `entity_log` 草案现写"RuleMatch 自行读取 EntityLog automation policy"，与本提案"经 handoff 投影、不直接读 EntityLog"不一致，待后续对齐（以 L2 提案为准）。
