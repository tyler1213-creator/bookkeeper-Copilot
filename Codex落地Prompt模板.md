# Codex 落地 Prompt 模板

## 文件角色

这是一份**可复用的元模板**。每当一个对象（workflow node 或 memory layer）的 L2 / M2 提案被用户审定后，由审查方（Claude）按本模板填空，生成一段发给 Codex 实现窗口的落地指令；Codex 据此对正式文档执行**改动 / 更新 / 创建**。

它不是某个对象的具体指令，本身不落地任何文档。每次使用时复制下方「模板正文」，把 `<…>` 占位符替换为该对象的真实内容，删掉不适用的分支，再发给 Codex。

一次只针对**一个对象**生成一份。跨对象改动（例如同一结论要同时改 A 和 B 两个对象的文档）拆成多份独立 prompt，分发到不同窗口，互不重叠，避免 git 冲突。

## 使用方式（给改编者 / Claude）

1. 先判定**模式**：到 `BK_Copilot/` 下查该对象正式草案是否存在。
   - 存在 → **改模式**：保留仍有效内容，只做提案 + 澄清要求的改动，不从零重写。
   - 不存在 → **建模式**：按 `BK_Copilot/node:layer spec template/` 模板从零建 00 / 01 / 02。
2. 判定对象类型：workflow node 用 `workflow_node_spec_template_rules.md`；memory layer 用 `memory_layer_spec_template_rules.md`。
3. 把审计中已收口的决策、用户口头审定的澄清，逐条填进「用户已审定的澄清」与「具体改动清单」。
4. 「具体改动清单」必须指到 `文件:章节`，不要让 Codex 自己猜要改哪。
5. 删掉模板里不适用的分支（如建模式删掉「改模式专项」段，反之亦然）。
6. 不确定某输出的 consumer / 某结论的归属时，在「停止并问用户」里列出，**不要替用户拍板写进指令**。

## 占位符清单

- `<对象名>`：如 Entity Log、Rule Match Node。
- `<模式>`：改 / 建。
- `<对象类型>`：workflow node / memory layer。
- `<文件清单>`：该对象在 `BK_Copilot/` 下的 00 / 01 / 02 路径。
- `<主驱动提案路径>`：`L2_proposals/<对象>__L2提案.md`。
- `<模板路径>`：对应的 node 或 memory layer 模板规则文件。
- `<用户已审定的澄清>`：凌驾于现状/提案措辞之上的口头锁定结论，逐条。
- `<具体改动清单>`：逐条 `文件:章节 → 改什么`，对应提案哪条决策 / 哪条澄清。
- `<相关规则文档清单>`：本次落地后需要同步更新的治理 / 跟踪文档（默认 `当前任务状态.md`、`缺口地图.md`，其余按需）。

# 模板正文（复制以下内容填空后发给 Codex）

```
# 任务：把已审定的 <对象名> L2/M2 提案落地到 BK_Copilot 正式文档（<模式>模式，仅此一个对象）

## 你的角色
你是实现者，不是审查者，也不是设计者。
唯一任务：把【已经过用户审定的】<对象名> 提案 + 下方用户澄清，落到 BK_Copilot 的对应正式文档上。
不重新设计、不重新审计、不质疑已锁结论、不扩大范围、不碰其它对象。

—— 若 <模式> = 改：目标文件已存在，保留仍有效的内容，只做提案与澄清要求的改动，不从零重写整份文档。
—— 若 <模式> = 建：目标文件不存在，严格按 <模板路径> 的结构从零创建 00_index / 01 / 02；只创建已成熟到值得写下来的文件，不为结构完整造空文档。

## 范围（严格）
本次允许编辑的对象文件，仅限：
<文件清单>

落地完成后，按本指令最后一节「落地后更新相关规则文档」同步更新：
<相关规则文档清单>

除上述文件外，绝不触碰其它 workflow node、memory layer、未列出的治理文档、new system/、old_system_nodedesign/。
如果驱动提案里出现「改其它对象文件」的拟改条目，【不要执行】——那属于另一个对象，由单独 prompt 处理；本 prompt 内对该结论只在自己范围的文件里声明。

## 必读（按此顺序，且只把它们当依据）
1. 主驱动：<主驱动提案路径> —— 本次改动/创建的权威来源，逐条决策落地。
2. 模板（必须遵守其结构与「禁止写法」）：<模板路径>
3. 现状/结构：
   - 改模式：上面列出的现有对象文件。
   - 建模式：<模板路径> 中 00_index / 01 / 02 的章节结构与各「完成清单」必答项。
4. 锁定上游/不变量：当前任务状态.md、缺口地图.md（出现冲突时以这两份已锁结论为准，按权威顺序解决，不以 new/old system 为准）。
new system/ 和 old_system_nodedesign/ 只是审计对象，没有任何权威性，不得作为依据或 baseline。

## 用户已审定的澄清（凌驾于现状文字与提案措辞之上；只存在于本指令，务必照此落地）
<用户已审定的澄清>
（逐条写明；每条澄清如与现状/提案某句冲突，点名要改的那句，给出最终口径。例如：状态两态、某接缝走投影而非自读、某归属归 X 不归 Y、不新增某字段等。）

## 不可违反的纪律
- 不以任何形式复活 candidate entity / candidate identity（含第三态、把 ambiguous/unresolved/conflict 当独立输出类别或独立身份状态保留）。
  注意：文档里现有的 candidate_signal、merge_split_candidate、automation/entity risk candidate、review/governance candidate、candidate profile fact 是合法的「非身份候选」（Alias/合并拆分/治理），保留它们；但它们不是身份状态，要与 stable/unknown 身份分类清楚分开。
- 不在 L2/M2 冻结字段级 schema：exact field schema、provenance 字段、enum 取值、阈值数字、refs 形态、存储/projection 形态、写入执行者/顺序，全部留作 open boundary（L3 或 L4/seam），不写死。现状里若已写死字段名，下沉为「字段名留 M3/L3」。
- 守运行/记忆 seam：
  - workflow node：只声明「要持久化什么 + 谁有权威认定有效」，不定「谁来写、怎么写、什么顺序写」。
  - memory layer：只定写入机制（怎么写/谁写/什么顺序），不重判「该不该写、内容对不对」（那来自发起写入的运行节点）。
- 契约面 / Consumer：
  - workflow node 的输出类别表，每个输出必须写明 Consumer（谁消费）与下游含义；缺 consumer 视为生成阻塞，停下问用户。
  - memory layer 的读取者/写入者表必须声明为对外接口面，reader 只依赖声明内容。
- 不触发模板「禁止写法」（如「LLM 自行判断」「高置信度自动通过/自动沉淀」「后续实现再定」「参考旧系统」「写入日志/trace 但不说哪个 store/字段/authority」「生成候选但不说谁消费/谁批准/能否持久化」）。

## 具体改动清单（逐条指到 文件:章节）
<具体改动清单>

—— 改模式专项：每条注明与现状的冲突点；关闭已被本轮收口的旧「未冻结」标记（00_index 未冻结边界 / Open Boundaries / 进入下一阶段）；不要留悬空引用。
—— 建模式专项：按模板补全 00_index（文档状态 / 覆盖阶段 / 当前未冻结边界 / 进入下一阶段前必须解决）、01（存在理由 / 唯一职责 / 排除范围 / 对核心目标贡献 / 约束 / 未决）、02（触发或 authority、读 / 写 / 权限、契约面与 consumer / reader、异常 / 冲突、trace、open boundaries）；对照模板「L2 / M1-M2 完成清单」逐条确保可回答，不能回答的标为 open boundary，不要猜。
—— 覆盖阶段：保持 Stage 1-2 / M1-M2，不勾选 Stage 3 / M3 及以后。

## 停止并问用户的情形
- 任一输出的 consumer / 某结论的归属无法从提案或已锁上游确定。
- 提案某条拟改与已锁上游（当前任务状态.md / 缺口地图.md）冲突，且权威顺序无法解决。
- 出现需要新的产品决策、而非措辞落地的点。
遇到以上情况，停下来问，不要自行拍板或猜测。

## 落地后更新相关规则文档（务必执行）
对象文档改完/建完后，同步更新本次落地涉及的规则 / 跟踪文档，使新窗口能据此接续：
- 当前任务状态.md（默认必更）：把该对象的 L2/M2 标为已落地；更新「已完成」「下一步」「最近交接」中的对应指针；若有新风险或待对齐项，记入「当前风险/最近交接」。只保留必要交接和指针，不搬长期结论全文。
- 缺口地图.md（默认必更）：把该对象已收口的项标为已收口、关闭已解决的 open boundary、更新仍挂起项（L3 / L4·seam / L2·外阻）的状态。
- 其它治理文档（系统上下文地图.md / 审计阶段路线图.md / AGENTS.md / 审计目标与原则.md / 项目背景速读.md）：仅当其各自职责范围内的内容确实因本次落地发生变化时才动；否则不要碰。

更新纪律：
- 保持克制，遵守权威顺序；规则文档不承载长期设计结论正文——长期结论的归属是正式 BK_Copilot 文档 / 对应主题记录，不是治理文档。
- 不在规则文档中加入产品功能逻辑细节，也不定义节点/字段/流程的好坏标准。
- 不把 new/old system 当 baseline 或权威。

## 完成后输出
- 改/建了哪些文件、每个文件改了哪些节、对应提案哪条决策或哪条澄清。
- 哪些点按要求保留为 open boundary（L3 / L4-seam / L2·外阻），没有越权冻结。
- 同步更新了哪些规则文档（当前任务状态 / 缺口地图 / 其它），各更新了什么；哪些规则文档判断无需改动及原因。
- 一致性自检：两态/无 candidate 复活、seam 守住、契约面与 Consumer/reader 已声明、未冻结字段级 schema、未触发禁止写法、覆盖阶段仍为 Stage 1-2 / M1-M2。
- 是否触发「停止并问用户」；若有，列出问题。
```
