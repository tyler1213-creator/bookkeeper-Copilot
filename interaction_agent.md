# 概念：交互 Agent / 目标层 Agent（Engagement Goal Agent，含覆盖完整性核对）

- **状态**：新创建概念（用户本轮明确要"单独的交互 Agent"）；圈外/未正式设计。
- **范围更新（2026-06-12）**：原独立概念"覆盖完整性检查 / 查缺补漏（coverage_completeness_check）"经讨论判定**不构成独立节点**，并入本 Agent 作为其一项职责。本 Agent 现在同时承担"前期问目标"与"后期核对齐不齐 + 缺了就问客户"。

## 一句话定义
在 bookkeeping 正式跑流程**之前**搞清这次 engagement 的目的（要做哪些客户、哪些账户、哪些期间），并在材料进来、去重之后**核对实际收到的覆盖是否齐全**，发现漏月/漏账户/多发重复就向客户提问。

## 为什么需要它
- "漏了没"必须有"该有多少"做标准，而这个标准**来自人，无法从材料本身推出**；需要一个角色把目标问清、记下来。
- "整个 engagement 齐没齐"是**跨批次、engagement 级**的问题：客户常分多次上传（先发账单、过几天再发小票；这周发支票户、下周发信用卡）。Evidence Intake 是**按批**运行的，单批跑的时候根本不知道缺的那个月会不会下次才来——所以完整性核对天然不能由 EI 单批输出，必须由一个握着 engagement 目标的角色在更高层做。
- "去问客户"本身也不是 EI 的活（EI 只暴露 issue、不向客户提问）；本 Agent 握着目标与客户对话通道，是发起提问的自然归属。

## 功能逻辑 / 如何运转

### 阶段一：前期问目标（建立期望清单）
1. 与客户/会计师交互，问清目标（例：完成 10 个客户一整年的账）；
2. 形成**期望清单**：按 **账户 × 期间** 展开（如 客户A·支票户·2024-01..12、客户A·信用卡·2024-01..12）。**不能用"文件数 = 月份数"**——一个客户常有多账户，3 张卡做满一年就是 36 个 (账户×月)。

### 阶段二：后期核对覆盖（去重之后）
3. 拿 Evidence Intake **去重后**输出的"实际唯一覆盖信号"（EI 实际收到、且未判为冗余的 账户×月份 集合）；
4. 与期望清单比对，两个维度互补查漏：
   - (a) **中段断口**：由 Evidence Intake 内部的**余额链**（见 EI 提案 2.4.1，`statement[n].期末 ≠ statement[n+1].期初`）确定性抓出，账单自暴缺口，不需要期望清单；
   - (b) **首尾漏月 / 整个账户没给**：余额链够不着（没有相邻账单可比、或该账户一张都没有），只能靠**期望清单点名**抓出；
5. 发现缺月/缺账户/多发重复 → 由本 Agent（或其客户对话通道）向客户提问，去要缺的、澄清多发的。

### 时机
- 完整性核对在**文件 hash + 账单/账期余额链去重之后**做：去重后才知道真实唯一覆盖；"总数对得上但有重复 = 实际缺月"。

## 边界（与已有对象划清）
- **不是 Profile**：Profile 管固定/周期交易记忆（账户名、按月还贷/发薪等）。
- **不是 Onboarding**：Onboarding 建客户身份背景，不做覆盖检查。
- **不是 Coordinator**：Coordinator 是流程后期、针对已收交易的疑点向客户提问。
- **不是 Evidence Intake**：EI 按批处理材料、只输出"实际覆盖"事实信号，不持有 engagement 目标、不下"齐没齐"结论、不向客户提问。
- 本 Agent 只管"这次要做到什么 + 据此查缺补漏 + 向客户提问"；它**检测并提问**，不擅自补造材料。

## 关系
- **上游**：客户/会计师（目标）；Evidence Intake（去重后的实际覆盖信号 + 余额链中段缺口信号）。
- 期望账户清单可能与 onboarding 已知账户有数据交集，但**拥有权在本 Agent**。
