### 问题 1：已确认 stable entity 后，当前 transaction surface text 是否以及如何进入 Alias 库？

问题已收窄。

Entity Resolution 同步写入 `new_stable_entity` 本体已确认。该写入只限 entity 本体，不同时写入 Alias、automation policy 或 rule。

本问题只讨论 Alias 写入，不讨论 automation policy。automation policy 后续放到 Rule Match / automation governance 相关讨论中单独处理。

已确认的 Alias 定义：

- Alias 是过去已经确认过的 transaction surface text 和 `stable entity` 的对应关系。
- 当前只确认两类信息可能成为 Alias：bank statement 中每笔交易的 `description` / `descriptor` / raw bank surface text；当 bank description 本身没有明确身份意义时，其他可能重复出现并能指向交易主体的字段，例如 cheque payee。
- 当前确认需要一个可被 Entity Resolution 查询的 Alias 库。
- Alias 只辅助 Entity Resolution 复用身份，不提供 rule、会计分类结论或自动落账权限。

已确认边界：

- `new_stable_entity` 写入 Entity Log 时，Entity Resolution 只写 entity 本体。
- stable entity 不等于当前 surface text 已经进入 Alias 库。
- stable entity 不等于拥有 active rule。
- stable entity 不等于拥有 automation policy。
- Rule Match 当前围绕 entity-level / pattern-level rule scope，不围绕 Alias 建立确定性自动化。

- Alias 写入是直接写入，还是进入统一 finalization / memory update 流程。
- Alias 写入是否需要额外 accountant approval，还是只要 surface text 与明确 stable entity 的关系已确认即可。

### 问题 2：Case Judgment 判断会计分类（COA）是否必须依赖已识别的 entity？（已解决）

**结论：** Case Judgment 的高置信度自动分类通道必须依赖已识别的 entity。Entity Resolution 输出 unknown_entity 时，Case Judgment 不走高置信度分类通道，但仍然处理这笔交易，输出 pending。

Pending 内部分两种情况：

1. **完全无法判断**：没有足够上下文支持推断，由 Coordinator 直接向 accountant 提问。
2. **有推断但不确定**：Case Judgment 利用已有的其他上下文（receipt items、交易金额模式等）给出推断性建议，由 Coordinator 呈现给 accountant 选择或确认。

三个子问题的回答：

- 是否存在不需要 entity 就能判断 COA 的交易？ → 存在（bank fee、interest 等），但属于 edge case，大部分由 Profile / Structural Match Node 在上游处理。到达 Case Judgment 的交易绝大多数需要 entity identity。
- unknown_entity 时 Case Judgment 能否用推测辅助分类？ → 不做高置信度分类，但可以给出推断性建议附在 pending 输出中供 accountant 选择。
- 基于不确定 identity 的高置信度分类后续如何修正？ → 不存在这个场景，因为不会产生高置信度分类。

A/B 分类判断（是否需要 entity 才能分类）的职责在 Case Judgment，不在 Entity Resolution。Entity Resolution 只管 identity。

详细推理链记录在 `Entity相关的所有问题.md` 的"Case Judgment 与 Entity Identity 的依赖关系"部分。

### 问题 3：Entity Resolution 输出 unknown_entity 时，是否应附带对 entity 的推测信息？（已解决）

**结论：** Entity Resolution 在无法识别出准确 entity 时，可以把所有与 identity 判断相关的证据、线索和判断说明作为 runtime context 传给 Case Judgment。

这些信息可以包括：

- 当前 evidence 中的人类可见身份线索。
- 为什么不能确认 stable entity 或 candidate entity。
- 可能的身份解释、相似对象、搜索结果或歧义来源。
- 支持或反驳这些解释的 evidence refs。
- Entity Resolution 对 identity gap 的判断说明。

使用边界：

- 这些信息不是 stable entity、candidate entity 或 identity authority。
- 这些信息不能写入 Entity Log，不能作为 Case Log 的 entity handle，也不能支持 Rule Match。
- 这些信息不能支持 Case Judgment 生成高置信度自动分类。
- Case Judgment 可以用这些信息判断 pending 子类型：完全无法判断，或有推断但不确定。
- Case Judgment 可以把这些信息转成 pending reason、missing information、question focus 和可选推断性建议，交给 Coordinator。
- Coordinator 可以用这些信息整理 accountant-facing question，但不能把推测包装成已确认事实。

详细结论记录在 `Entity相关的所有问题.md` 的"Unknown Entity 时 Entity Resolution 给 Case Judgment 的信息边界（已确认）"部分。

### 问题 4：Entity Resolution 输出 unknown entity 后的处理路径应该如何设计？（已解决）

**结论：** 当 Entity Resolution 输出 `unknown_entity` 时，交易进入 Case Judgment 后输出 pending。Coordinator 根据 pending 子类型设计结构化问题向 accountant 提问。Accountant 的回复在当前 batch 内同步处理。Accountant 明确确认 entity identity 后直接创建 stable entity，不需要 governance approval。交易不重新进入 Entity Resolution，也不重新进入 Case Judgment。

路径总览：

```text
Entity Resolution → unknown_entity
    ↓
Case Judgment → pending（保留两种子类型标记）
    ↓
Coordinator 根据 pending 子类型设计问题
    ↓
accountant 回复（当前 batch 内同步）
    ↓
创建 stable entity + 完成分类（或追问直到充分）
    ↓
写入各 log（机制由问题 7 决定）
```

两个 pending 分支：

1. **有推断但不确定**：Case Judgment 输出 pending 并附带推断性建议；Coordinator 同时询问 entity identity，并让 accountant 确认或修改分类。
2. **完全无法判断**：Coordinator 第一轮同时收集 entity identity 与业务用途 / 分类相关信息；如果回复不足，继续追问直到信息充分。

已确认边界：

- unknown entity 时不走高置信度自动分类。
- 不重新进入 Entity Resolution；accountant 的明确身份确认替代身份判断。
- 不重新进入 Case Judgment；分类在 Coordinator 与 accountant 的交互中完成。
- 不依赖 entity 的交易类型（如 bank fee、interest 等）属于 edge case，大部分由 Profile / Structural Match Node 上游处理；到达 Case Judgment 后按 pending 路径处理。

剩余依赖：

- Coordinator 的具体问题模板、回答结构和追问边界继续放在问题 5。
- Stable entity 创建执行机制与多 log 写入机制继续放在问题 7。

详细推理链记录在 `Entity相关的所有问题.md` 的"Unknown Entity 后的完整处理路径（已确认）"部分。
