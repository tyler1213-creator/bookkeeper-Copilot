# 系统上下文地图

## 文件职责

`系统上下文地图.md` 是新窗口理解系统材料的速读地图。

一句话职责：告诉新窗口“材料大概怎么分层、哪些来源有效、哪些目录只是审计对象”。

它只负责材料导航，不定义 node、field、memory layer 或 workflow 的正式边界，不替代 `BK_Copilot/`、`缺口地图.md`、主题记录或用户指定的长期结论文档。

## 当前使用规则

如需判断操作纪律、authority 或停止条件，读 `AGENTS.md`。

如需判断当前任务、下一步或 active risks，读 `当前任务状态.md`。

如需判断某个 node / memory layer 的当前正式边界，读 `BK_Copilot/` 下对应正式草案。

如需查看尚未收口的问题，读 `缺口地图.md`；该文件只索引未冻结 L3 / L4 / seam / L2·外阻缺口。

以下目录不进入 source authority 顺序：

- `new system/`
- `old_system_nodedesign/`

它们只作为审计对象。读取它们只能帮助理解材料如何表达自己，不能用作权威参考、正确性证明、baseline、历史意图权威或可复用约束来源。

## New / Old 材料粗略导航

New system 材料大体声称围绕这条交易处理链路展开：

```text
raw evidence
-> transaction identity
-> profile / structural check
-> entity resolution
-> rule match
-> case judgment / pending / review
-> JE generation
-> transaction logging
-> case memory / governance / knowledge update
```

Old system 材料更偏 pattern-centered automation：

```text
raw files
-> parsed transaction data
-> canonical description / pattern
-> profile match
-> rules exact match
-> confidence classifier
-> coordinator / review
-> observations / rules / transaction log updates
```

这些只是定位被审计材料的粗略地图，不是冻结 contract，也不是正确性证明。具体结论必须回到正式草案、主题记录、用户指定的长期结论文档或用户明确指令。

## 使用边界

正确用法：

1. 用本文件快速知道系统材料分布。
2. 回到 `当前任务状态.md` 确认当前任务。
3. 按任务读取 `BK_Copilot/`、`L2_proposals/`、主题记录，或在需要审计 source text 时读取 `new system/` / `old_system_nodedesign/`。
4. 把未冻结缺口写入 `缺口地图.md` 或对应正式草案的 Open Boundary。
5. 把长期性结论写入用户指定的长期结论文档或对应正式草案，不写进本摘要。
