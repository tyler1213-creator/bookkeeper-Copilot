# 系统上下文地图

## 文件职责

`系统上下文地图.md` 是新窗口理解系统材料的速读地图。

一句话职责：告诉新窗口“材料大概怎么分层、哪些来源有效、哪些目录只是审计对象”。

它只负责：

- 给出 New system / Old system 材料声称的粗略流程地图
- 标明哪些材料是正式草案、主题记录或被审计材料
- 帮助 agent 快速找到该读的源材料

它不负责：

- 定义 node、field、memory layer 或 workflow 的正式边界
- 记录长期性设计结论
- 替代 `BK_Copilot/`、主题记录或用户后续创建的长期结论文档
- 证明 New system 正确
- 把 Old system 当 baseline、历史意图权威或参考依据
- 给 `new system/` 或 `old_system_nodedesign/` 中的任何信息赋予 authority

## 当前使用规则

如需准确判断当前有效规则、正式草案或已确认结论，按这个顺序找源材料：

1. 用户明确指令。
2. `BK_Copilot/` 下已经重写的正式 node / memory layer 草案。
3. 当前主题记录，例如 `Entity相关的所有问题.md`、`Alias question.md`、`Rule Match临时文档.md`、`Coordinator Question.md`、`未解决问题暂存清单.md`。
4. 用户后续创建或指定的长期结论文档。
以下目录不进入 source authority 顺序：

- `new system/`
- `old_system_nodedesign/`

它们只作为审计对象。读取它们只能帮助理解材料如何表达自己，不能用作权威参考、正确性证明、baseline、历史意图权威或可复用约束来源。

如果本摘要和上述源材料冲突，以源材料为准。

## 新系统粗略流程

New system 材料大体声称围绕这条交易处理链路展开：

```text
raw evidence
-> transaction identity
-> profile / structural check
-> entity resolution
-> rule match
-> case judgment / pending / review
-> JE generation
-> transaction logging
-> case memory / governance / knowledge update
```

这只是被审计材料的导航图，不是冻结 contract，也不是正确性证明。

需要具体 node 边界时，先找 `BK_Copilot/` 中是否已有正式草案；没有草案时，读取 `new system/node_stage_designs/` 中对应 node 文件只能作为审计对象取样，不能把其中内容当成当前 authority。

## New / Old 材料粗略差异

Old system 材料更偏 pattern-centered automation：

```text
raw files
-> parsed transaction data
-> canonical description / pattern
-> profile match
-> rules exact match
-> confidence classifier
-> coordinator / review
-> observations / rules / transaction log updates
```

New system 材料更偏 evidence / identity / memory / governance 分层：

- 先保留可追溯 evidence。
- 再处理 transaction identity 和业务对象 identity。
- 再区分 rule、case judgment、pending、review、logging 和 governance。
- 不把 Old system pattern 或 New system 分层直接当成当前身份、规则或产品 authority。

这些差异只是帮助定位被审计材料；具体结论必须回到正式草案、主题记录、用户指定的长期结论文档或用户明确指令确认。

## 使用边界

审计新系统时，不要只凭本摘要下结论。

正确用法是：

1. 用本文件快速知道系统大概有哪些部分。
2. 回到 `当前任务状态.md` 确认当前任务。
3. 按任务读取 `BK_Copilot/`、主题记录，或在需要审计 source text 时读取 `new system/` / `old_system_nodedesign/`。
4. 把长期性结论写入用户指定的长期结论文档或对应正式草案，不写进本摘要。
