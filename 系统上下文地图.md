# 系统上下文地图

## 文件职责

`系统上下文地图.md` 是新窗口理解系统材料的速读地图。

一句话职责：告诉新窗口“这个系统大概怎么分层、要去哪里看更权威的细节”。

它只负责：

- 给出新旧系统的粗略流程地图
- 标明哪些材料是当前系统、正式草案、主题记录或历史参考
- 帮助 agent 快速找到该读的源材料

它不负责：

- 定义 node、field、memory layer 或 workflow 的正式边界
- 记录长期性设计结论
- 替代 `BK_Copilot/`、主题记录或用户后续创建的长期结论文档
- 证明当前系统正确
- 把旧系统当 baseline

## 当前使用规则

如需准确判断系统细节，按这个顺序找源材料：

1. 用户明确指令。
2. `BK_Copilot/` 下已经重写的正式 node / memory layer 草案。
3. 当前主题记录，例如 `Entity相关的所有问题.md`、`Alias question.md`、`未解决问题暂存清单.md`。
4. 用户后续创建或指定的长期结论文档。
5. `new system/` 下的当前系统设计材料，作为审计对象。
6. `old_system_nodedesign/` 下的旧系统材料，作为历史意图参考。

如果本摘要和上述源材料冲突，以源材料为准。

## 新系统粗略流程

当前系统大体围绕这条交易处理链路展开：

```text
raw evidence
-> transaction identity
-> profile / structural check
-> entity resolution
-> rule match
-> case judgment / pending / review
-> JE generation
-> transaction logging
-> case memory / governance / knowledge update
```

这只是导航图，不是冻结 contract。

需要具体 node 边界时，先找 `BK_Copilot/` 中是否已有正式草案；没有草案时，再读 `new system/node_stage_designs/` 中对应 node 文件。

## 当前材料地图

- `new system/new_system.md`：当前系统总览，作为审计对象。
- `new system/node_stage_designs/`：当前 node 设计材料，作为审计输入。
- `BK_Copilot/workflow_nodes/`：已经开始重写的 workflow node 正式草案。
- `BK_Copilot/memory_layers/`：已经开始重写的 memory layer 正式草案。
- `BK_Copilot/memory_layers/alias_log/`：Alias Log 正式草案，记录已确认 transaction surface text -> stable entity 的身份复用边界；具体技术形态尚未冻结。
- `BK_Copilot/memory_layers/case_log/`：Case Log 正式草案，记录以 entity 为主要索引的 completed-case learning memory 边界；当前覆盖 M1-M2，不冻结 data contract。
- `BK_Copilot/node:layer spec template/`：后续写正式 node / memory layer spec 的模板规则。
- `Entity相关的所有问题.md`：当前 Entity 讨论记录。
- `Alias question.md`：Alias 主题速记。
- `未解决问题暂存清单.md`：用户明确同意暂存的问题。
- `old_system_nodedesign/`：旧系统设计意图参考。

## 新旧系统粗略差异

旧系统更偏 pattern-centered automation：

```text
raw files
-> parsed transaction data
-> canonical description / pattern
-> profile match
-> rules exact match
-> confidence classifier
-> coordinator / review
-> observations / rules / transaction log updates
```

新系统更偏 evidence / identity / memory / governance 分层：

- 先保留可追溯 evidence。
- 再处理 transaction identity 和业务对象 identity。
- 再区分 rule、case judgment、pending、review、logging 和 governance。
- 不把旧系统 pattern 直接当成当前系统的身份或规则 authority。

这些差异只是帮助定位材料；具体结论必须回到正式草案、主题记录或用户指定的长期结论文档确认。

## 使用边界

审计新系统时，不要只凭本摘要下结论。

正确用法是：

1. 用本文件快速知道系统大概有哪些部分。
2. 回到 `当前任务状态.md` 确认当前任务。
3. 按任务读取 `BK_Copilot/`、主题记录、`new system/` 或 `old_system_nodedesign/`。
4. 把长期性结论写入用户指定的长期结论文档或对应正式草案，不写进本摘要。
