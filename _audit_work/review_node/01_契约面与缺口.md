## 已锁事实（遵守，非审查对象）

- **Coordinator/Review 分界已锁**：running→Coordinator，finalized+人发起→Review。（Coordinator 02 §1）
- **运行层四 Log 分界已锁**（Transaction Log 02 §7）：交互事实→Intervention Log；身份→Entity Log；可复用先例→Case Log；最终交易审计→Transaction Log。
- **Transaction Log append-only**（外部审计强制，可当公理）。
- **rule 无自动成立 / 降级**：一切 rule authority 变更必须 accountant sign-off + Governance（Rule Log 02 §3/§5；Entity Log 02 §3 automation upgrade 须 approval，仅 restrictive auto-downgrade 可受控自动且须治理可见）。
- **Review 在 Entity Log / Case Log 的角色已被 A 命名**：Review = 这两层的合法 candidate 产出方 / reader，但**绝不直接写 stable authority / 不把案例变 rule**。

## △缺口（A 锁不住、要交用户判断的开放问题 —— 本步最值钱产物）

- **△1 Intervention Log 无正式 spec**：Coordinator 02 §12、Transaction Log 02 §7/OpenB 均把 Intervention Log 列 L2·外阻（schema/writer/边界未落文）。Review 要「生成 Intervention ID 链路、记为什么改」——其落点 Log 本身未落文。→ 挂 open boundary，**绝不拿任何材料填**。
- **△2 Governance Log 不存在**：doc 自述「当前不存在，必须新建」。Review 高半径/永久纠错的审批审计账本无 A 类锚。→ 挂 open boundary，新建属另一对象的审计范畴。
- **△3 `confirmed_by` 字段名/语义未锁**：Case Log 02 §10 列 `use_level`/`confirmed_by` exact enum 为 M3 缺口。Review 第 4 节「按先例确认来源过滤作废范围」依赖该字段。→ 与 Case Log M3 对齐前是 △。
- **△4 entity 税务状态字段是否存在未知**：Entity Log 02 §10 五类信息 field schema 留 M3，未确认是否记录税务状态字段。Review 「改分类⇒重算 HST/GST」是查表还是判断，取决于此。→ △，缺失时按「引擎撞缺失输入→停下进 read-back」。
- **△5 「rule 被推翻一次」信号的落点**：Rule Log 02 把 rule health/staleness 派生统计列 L4/seam，且 rule 无自动降级。Review 发的「推翻一次」信号去「治理/发现层」——该发现层（Post-Batch Lint / Governance Review）正式边界未落文。→ △ / L2·外阻。
- **△6 merge/split 发起权 A 已部分锁定**：Entity Log 02 §3 已把 Review 列为 `merge_split_candidate` 产出方（人发起，candidate-only）。doc Open Boundary #5「语义发现器是否并入 Review 由人发起」与此**方向一致**，但「系统自发 merge/split」是否还需要 = 真实产品决策，已挂记忆 `semantic-discovery-node-necessity-open`，另开窗口。→ 主线程对齐项。
- **△7 影响展开引擎依赖图的声明位置/形态**：doc 第 8 节自承属 L4/seam 待定；无 A 锚。→ L4/seam。
- **△8 统一 finalization / 授权确认机制 exact 实现**：跨 log 原子/顺序/幂等机制，Transaction/Case/Entity Log 全部列 L4/seam 未冻结。Review 只能调用、不重定义。→ 守 seam。

## 语义对齐待主线程裁决（只摆两端，不在此下结论）

- doc 第 4 节「把对应旧先例标记 superseded → Case Log」vs Case Log 02 §5「invalid/superseded 由 correction/finalization/governance result 触发，机制未冻结」「不动人工确认过的先例」——语义方向一致，**机制未冻结**，且「人工确认 vs 系统确认」依赖 △3 字段。
- doc 第 2 节三层（理解器 LLM / 影响展开引擎 / 确定性写手）的「确定性写手」=「统一 finalization 写入机制」（共享，跨 log）——与 Transaction Log 02 §3「统一 finalization 写入机制（exact writer 未冻结）」同名同义，Review 不得重定义其机制。
