# 02 热身与防污染（B 类，用完即弃 · 不具权威）

> 来源：`new system/node_stage_designs/coordinator_pending_node__design.md` [待审计·不具权威]。仅热身 + 问题发生器，绝不继承字段名/枚举/状态机/输入输出形状。

## 1. 行话 / 领域速记（帮主线程快速进入状态）

- B 把节点叫 "Coordinator / Pending Node = runtime clarification and intervention bridge"：交易因证据/身份/结构/rule/case/authority 问题不能安全继续时介入，向 accountant 提聚焦问题并保留回答语境。
- B 的核心机制：把上游卡点（pending reason）转成 accountant 能回答的问题；保留 intervention 语境；产出多种 candidate handoff；hard block 不能被提问解除。
- B 反复强调的边界（与 A 方向一致的部分）：不写 Transaction Log、不做最终分类/HST/JE、不直接改 Profile/Entity/Case/Rule/Governance、candidate 不等于 approval、模糊回答不能包装成 confidence。

## 2. B 触及、值得问用户的候选问题（只列问题，不给答案）

- B 把 Coordinator 触发源画得很宽（Evidence Intake / Transaction Identity / Profile / ER / Rule Match / Case Judgment / review_context / workflow_orchestrator 都能触发）。**问用户**：当前系统 Coordinator 的输入是否只收 CJ 的 Pending（A 锁「Pending 唯一 consumer = Coordinator」），还是也直接收其他节点卡点？（注：Transaction Identity 节点已删除——见当前任务状态.md。）
- B 设了 `Intervention Log` 作为 Coordinator 唯一 durable write。**问用户**：当前是否要 Coordinator 持有一个 durable interaction record？归属哪个 log？（A 类无 Intervention Log 正式 spec → 圈外。）
- B 列了一大堆 candidate handoff（profile/entity/alias/role/rule/automation-policy/case/governance/review）。**问用户**：Coordinator 当前到底允许产出哪些 candidate？（A 类对 Coordinator 的 candidate 权限未单独成文；Coordinator Question.md 给的是「确认身份触发 Entity Log 写入」这一具体 authority。）
- B 有 `runtime_continuation_handoff`（回到上游/下游受限重跑）。**问用户**：accountant 回答后交易往哪走、是否重入某节点、重跑边界？（A 标未冻结。）
- B 把问题聚合/去重列为 open boundary。**问用户**：answer-time 去重（同 surface 合并提问、答一次扇出）是否纳入本轮 Coordinator？（Coordinator Question.md 已有倾向，暂存清单问题 6 暂缓。）

## 3. 防污染小抄（B 与 A 明显分叉处，以 A 为准）——主线程护栏

- **【别顺 B 走】B：Coordinator 唯一 durable write = Intervention Log，且明文「不能 create stable entity」。**
  以 A 为准：Entity Log §3 设有「Accountant explicit identity confirmation path」写入者，**Coordinator 路径在 accountant 明确确认身份后触发 stable entity 创建（无需 governance approval）**（Entity Log §3/§6 + Coordinator Question.md）。→ 这是 A 与 B 的实质冲突；按铁律以 A + 用户已拍板的 Coordinator Question 结论为准。**这是停止/报告点之一，已在 01 接缝 3 标注。**
- **【别顺 B 走】B 的全套字段/枚举**（`pending_coordination_request`、`pending_issue_context.issue_type` 20 个枚举、`authority_limit_context.hard_blocks`、各 output 的 `*_status` 枚举等）。一律不得继承——字段级 schema 属 Stage 3，且必须从 A 重建（CJ Pending handoff 最小语义 + ER candidate_signal + Entity Log/Case Log 边界）。
- **【别顺 B 走】B 含已删除对象**：`Transaction Identity Node` 作为触发源、`pattern_source` 等。当前系统 Transaction Identity 已删除（transaction_id 并入 Evidence Intake，当前任务状态.md）。
- **【别顺 B 走】B 把触发源画得极宽**（含 workflow_orchestrator、review_context 直接触发）。A 只锁「CJ Pending 的唯一 consumer 是 Coordinator」+ ER candidate_signal 列 Coordinator 为 consumer 之一。其余触发源在 A 无依据 → 不得据 B 补。
- **【别顺 B 走】B 的 `role` / role confirmation candidate 概念**。Coordinator Question.md 明确「`Role` 字段此前已删除，本轮不重新启用」。
