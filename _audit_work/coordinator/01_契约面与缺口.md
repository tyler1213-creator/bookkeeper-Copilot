## 接缝 1 — Case Judgment → Coordinator（最核心，直接上游）

A 原文：
- 「Coordinator：Pending 的唯一 consumer，负责把 reason、候选建议、交易/entity 信息组织成会计师可回答的问题。」（CJ 01 §4 下游）
- Pending 输出表 Consumer = 「Coordinator（唯一 consumer）」；下游影响：「Coordinator 负责把 reason、候选、交易/entity 信息组织成会计师可回答的问题；会计师确认且信息足够后，由 Coordinator 路径转 JE 所需格式或继续追问。」（CJ 02 §6）
- **Pending handoff 最小语义（CJ 02 §6）**：Pending 必须至少表达 ① 客户/identity 背景；② 相关 Case Log 先例摘要（仅 stable entity 时存在）；③ 阻塞原因（卡在哪、为什么不能判断/放行）；④ 已执行工作及结果（如已联网但无确定结果）；⑤ 可选推断性建议和不确定项。reason 只解释为何 Pending、定位断点，不开补救药方、不作下游分流开关。
- Pending 触发面（CJ 02 §6 / §7-§9）：hard block、identity unknown、证据不足、HST/GST 不确定、多合理候选、冲突无法干净解决、证据明确指向非公司支出需会计师立场、COA 校验重判仍失败。
- 「Coordinator 负责人工交互，不应在上下文不全时重做 CJ 的会计分析。」（CJ 01 §4）

△缺口：
- Pending handoff 的 exact field schema / `pending_request_context` — CJ 标 L3（CJ 02 §12.3；缺口地图 CJ section）。Coordinator 输入契约的精确字段随之未定。
- 「会计师确认且信息足够后，由 Coordinator 路径转 JE 所需格式」——转 JE 格式 contract 未定（CJ 02 §12.15；缺口地图）。
- 「继续追问」的追问结构 / 问题模板 / 候选组织机制未定（CJ 02 §12.15）。

## 接缝 2 — Entity Resolution → (CJ) → Coordinator（unknown 身份路径）

A 原文：
- ER `unknown` 输出表 Consumer = Case Judgment（输出 pending）；并「可读取身份线索、搜索线索、相似对象、reason 和 evidence refs 作为 runtime context」。（ER 02 §6）
- ER `candidate_signal` 输出表 Consumer = 「Coordinator / Review / Case Memory Update / Governance Review」，「只作为 runtime handoff 供对应 consumer 进入 pending、review、case memory update 或 governance review 判断」。（ER 02 §6）
- **Problem 2 结论（ER 02 §7）**：unknown 后 pending 内部分两种——「完全无法判断」由 Coordinator 直接向 accountant 提问；「有推断但不确定」CJ 给推断性建议附在 pending，由 Coordinator 呈现给 accountant 选择或确认。
- **Problem 4 结论（ER 02 §7）**：unknown 后 → CJ pending → 「Coordinator 根据 pending 子类型向 accountant 提问」→「Accountant 明确确认 identity 后，交易不重新进入本节点（ER）」「已确认交易不重新进入 Case Judgment；分类在 Coordinator 与 accountant 的交互中完成」。「Accountant confirmation 替代 Entity Resolution 的身份判断；stable entity 创建和最终分类由后续路径完成。」「Coordinator 追问结构、问题模板和 finalization / memory write 机制仍属于下游未冻结问题。」

△缺口：
- 「分类在 Coordinator 与 accountant 的交互中完成」「已确认交易不重入 CJ」——Coordinator 在身份确认后如何承接/产出最终分类（自己不做会计分类，但要把 accountant 答复转成结果），其路径与 CJ 「Coordinator 不重做会计分析」如何不矛盾，需主线程对齐 + 用户判断。
- pending「子类型」枚举未定（ER 称「根据 pending 子类型」，但子类型 schema = CJ L3）。

## 接缝 3 — Coordinator → Entity Log（accountant 确认身份后创建 stable entity）

A 原文：
- Entity Log §3 写入者：**「Accountant explicit identity confirmation path（会计师明确身份确认路径，执行者留 L4/seam）」**，可写「stable entity 本体和最小创建 provenance（字段名留 M3）」，写入类型「direct stable entity creation via finalization / memory write mechanism（实际机制未冻结）」，**「不需要 governance approval；只确认 identity，不隐含分类结果、Alias、Rule、automation_policy 或 Case Log 写入」**。（Entity Log §3、§6 第二条 mutation path）
- Entity Log §2 reader：「Coordinator / Pending Node」读「blocking context、risk flags、candidate refs」，用途「将 Alias / identity gap 转成人工问题」，限制「accountant answer 不能由本节点直接写入 Entity Log」。（Entity Log §2）
- Entity Log §6 约束：「Accountant 只确认会计分类、没有明确确认 identity 时，不创建 stable entity，也不写 Entity Log。」
- Entity Log §1：entity-centered Alias 的 source of truth 在 Entity Log；Alias Log 是 alias→entity 反查 projection；「Entity Log 与 Alias Log 在 merge/split 治理时需要同步更新；同步执行机制留 L4/seam」。（Entity Log §1、§4、§7）
- **Coordinator Question.md（用户指定本轮入口）**：「主体确认发生在 Coordinator 路径。… accountant 明确回复主体身份后，该主体才被确认为 stable entity。此时由 Coordinator 路径发出 entity 确认信号，触发 Entity Log 写入（设计意图：与 Alias Log 同步写入）。」「由 Coordinator 亲自写、还是同步调用 Entity Log writer / 统一 finalization 机制，以及 Entity Log 与 Alias Log 的同步写入 exact 机制，均未冻结（Entity Log Open Boundary，L4/seam）。」

△缺口（关键，多为已锁语义 + L4/seam 机制）：
- 「Coordinator 路径发出 entity 确认信号」的执行者 / 调用方式 / 与 Alias Log 同步写入机制 = L4/seam（Entity Log §6、§10.4/.6；Coordinator Question.md）。语义已锁（无需 governance、只确认 identity），机制未冻结。
- §2 reader 限制写「accountant answer 不能由本节点直接写入 Entity Log」与 §3「Accountant explicit identity confirmation path」写入者并存——二者是「Coordinator 不直接写、但确认路径触发写入」的 seam，需主线程对齐确认（这是 A 内部两条原文，需读成一致）。

## 接缝 4 — Coordinator/accountant 交互产物 → Case Log / Transaction Log（情况 Y 身份缺口）

A 原文：
- Case Log §7 Transaction Log 行：「entity 未解析即完成分类的交易只 finalize 到 Transaction Log，不进入 Case Log」；未冻结含「身份缺口标记落点（初步倾向 Transaction Log）」。
- Case Log §3 写入者：「Review / Coordinator / accountant interaction finalization path（未冻结）」可作「accountant-confirmed outcome 或 correction context 进入 Case Log 的来源上下文」，「具体写入机制未冻结」。
- **Coordinator Question.md**：情况 Y（主体不明）「留下一个不阻塞的身份缺口；其记录边界见 case_log/：只进 Transaction Log，不写 Case Log/Entity Log，不阻塞交易，不构成 candidate entity」。情况 X（不需要主体）「没有身份缺口，unknown 是正确终态」。
- Coordinator Question.md 边界：「Coordinator 不把 accountant 的模糊回答包装成已确认身份」「不替代 Case Judgment 做自动分类」「与 accountant 的交互本身不构成 Entity Log / Case Log / Rule Log authority」。

△缺口：
- 情况 X / 情况 Y 的显式收集（追问「是你也不知道主体，还是即使知道也不重要？」）属 Coordinator 行为，已在 Coordinator Question.md 确认；但其在 question 模板中的落点 = 未冻结（暂存清单问题 5）。
- 情况 Y 身份缺口待办由谁 triage / 复查机制 = 用户暂缓（Coordinator Question.md §仍未冻结）。

## 接缝 6 — Rule Match（弱关联）

A 原文：Rule Match `rule_miss` → CJ（CJ 01 §4 / 02 §1）。Rule Match 不直接对 Coordinator。RuleMatch L2 未提 Coordinator 直接接缝。
△缺口：无直接接缝；blocked / governance-needed 是否经 CJ→Pending→Coordinator 还是另路，未在 A 明确（属圈外编排）。

## 接缝 7 — 已锁事实清单（遵守，非审查）

- 身份判断 `stable`/`unknown` 锁在 ER；会计分类归 CJ；身份缺口记录边界 Case Log/Transaction Log。
- accountant identity confirmation 创建 stable entity 无需 governance approval、只确认 identity（Entity Log §3）。
- automation policy upgrade/relaxation 必须 accountant/governance approval；自动降级须 governance visibility（Entity Log §1、§5；CJ 02 §5）。
- AI 联网身份用途锁 ER、会计事实用途锁 CJ；搜索摘要只作 clue。
