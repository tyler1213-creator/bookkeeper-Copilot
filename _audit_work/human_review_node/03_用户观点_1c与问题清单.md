# 03 · 用户观点（1c）+ 修正的 writer 模型 + 问题清单（已分类）

## 0. writer 模型修正（owner 本轮确认 —— 唯一的"其他节点"改动）

- **其他节点的功能逻辑与 authority 全部不变。** 唯一变化是「谁来写」。
- **旧模型（已废）**：节点发 candidate signal → 某指定节点接收 signal → 执行写入。
- **新模型（现行）**：每个节点配一段确定性代码机制；**节点自己备好写入材料 → 调用该代码 → 由代码对相应 Log 精准写入**。
- 影响：`01` 重建的契约面（authority 面）**全部仍成立**；正式 log 文档里"candidate→接收节点写入"的描述属**待写回的 seam 级旧账**，不阻塞 Review。Transaction Log 02 §3 已指向的"统一 finalization 写入机制"正是此新模型的落点。

## 1. Review Node = 一个节点，内含 8 项（owner 明确：全留节点内，非平级机制）

1. **复核视图**：按 Bank Statement 聚合展示当天/过往已自动完成（High-Confidence + Rule 产出）交易。既是事后查看处，也是纠错入口。（非 LLM）
2. **Change List LLM（理解+提案）**：听懂自然语言纠错 → 结构化纠错意图（指哪笔/改成什么/scope 多大）；撒大网产出一份 Change List；**只提案、永不写字节**。
3. **Change List Engine（影响展开）**：沿设计期声明好的依赖图遍历，独立产出一份 Change List（不变量型后果：改分类⇒重算税/作废先例/改 source/生成更正记录）。**留在 Review 内**（当前唯一使用者就是 Review）；裸遍历未来若被治理流复用再议。
4. **对账 + 仲裁**：两份 Change List 比对（都有→留；只 Engine→信 Engine；只 LLM→进 read-back 让会计师判）。节点内一个执行步骤。
5. **落盘前 read-back + 三铁律**：每笔纠错把对齐后清单用人话念给会计师确认；① LLM 永不写字节 ② read-back 强制不分大小 ③ scope 绝不静默升级（"改这一笔" vs "以后永远这么走" = 两次单独确认）。
6. **单笔纠错级联（低半径）**：确定落点 —— Transaction Log append 更正记录；Intervention Log 记原因+生成 ID 回写；Case Log 把 `confirmed_by=system` 旧先例标 superseded（**不动 `confirmed_by=accountant`**）；source 改为 Accountant；若来自 active rule 则发"rule 被推翻一次"信号。**低半径不过授权闸。**
7. **rule 坏了 vs 一次性例外**：Review 里唯一真判断，但**降级** —— 单笔纠错时 LLM 不当场裁决，只做结构化纠正 + 发信号；"坏没坏"由攒证据的过程另算。
8. **永久纠错（高半径）出口**：会计师显式"以后都这么走" = 撰写 rule → 交节点外**授权确认机制**（先互斥校验，再人签字落盘）。

**净结构**：一个 Review Node。其中 6 的落盘、8 的审批，都是**调用节点外机制**完成，不在节点内重复实现。

## 2. 节点之外、另窗细化的三个机制（Review 只挂起引用，绝不重定义/填充）

1. **确定性发现**：定时/批后确定性代码扫历史（Case Log 等）→ 捞阈值型候选（rule 升级资格、automation 漂移、引用/互斥校验）→ 交授权侧。已定：确定性、非 LLM、非节点（是 job）、只发现不落地不审批；**语义型 merge/split 不归它**（归被挂起的语义发现器问题）。未定：调用/节奏、"够格"阈值（与 Rule Log promotion 资格耦合）。
2. **Finalization 写入机制**：唯一真正写各 log 的确定性代码；跨 log 原子/顺序/幂等/强制不变量/审计 trace。所有节点调它写。已定：确定性、非 LLM、共享、L4/seam、纠错与审批共用；各 memory layer 的"谁写/什么顺序" open boundary 应指向它。未定：调用契约、写入顺序、原子/回滚、幂等键。
3. **授权确认机制**：所有扩张型长期权威变更通往写入机制的**唯一、强制、不可旁路**关卡。本身不判断（判断者是会计师），做 read-back + 承载对账仲裁 + 接签字 + 带凭证触发写入。已定：共享、非节点；低半径单笔不经过它，高半径/永久/rule 变更/merge-split/automation 放宽必经过；写入机制只认带凭证请求。未定：在哪/被谁/何时调用、"扩张型变更"精确边界、凭证形态、与 Governance Log 写入耦合。

> **Governance Log**：三者共用的审计**数据层**，当前缺件、需新建；是数据层不是机制，不混入 Review 设计。

## 4. 问题清单（以"要按模板写成正式 node spec 还差什么"为标尺，已分类）

### L1/L2（动笔前必须讨论）
- **O1 scope/身份**：本节点是否**仅**事后(post-finalization)纠错？与旧 new_system「Review Node」(JE 前的 accountant gate, node#9) 及 A 类文档里"Review 覆盖 final logging 前交易"的引用是什么关系？（命名 + 边界）
- **O2 复核视图聚合范围/触发**：精确展示什么——当天/过往；High-Confidence + Rule 产出；是否含 Pending 解决后、terminal/blocked、Coordinator 内 accountant 确认过的？触发是会计师主动进入还是有调度。
- **O3 diff 成本分层**（doc OB#1）：每笔纠错都跑 LLM 第二意见 + 对账，还是按半径/新颖度分层（低半径 Engine 单跑、高半径/新型才 diff）？diff 机制本身保留已确认，触发分层未定。
- **O4 merge/split 发起权**：A(Entity Log 列 Review 为 merge_split_candidate 产出方) vs 1c(语义 merge/split 挂起、另窗) 张力——人发起的 Review 是否产出 merge/split 候选？
- **O5 HST/GST 重算性质**（△4）：查表(依赖 entity 税务状态字段) vs 判断；字段缺失时按"引擎撞缺失输入→停下进 read-back、不交 LLM 猜"。行为规则可现在定；字段存在性属 Entity Log L3/外阻。
- **O6 "rule 坏了"攒证据 与 确定性发现 job 的边界**：Review 侧只发"推翻一次"信号、不裁决；信号落点 + 累积判定归确定性发现/Rule Log promotion（外）。确认 Review 侧职责到此为止。
- **O7 高半径出口契约**：Review 交给授权确认机制的是什么（扩张型变更提案 + 对账后 Change List + read-back 结果）？守 seam，exact division pending 授权确认机制（另窗）。

### L3（字段/enum，留 Stage 3）
- confirmed_by exact enum（Case Log M3）；Intervention ID / correction record schema；Change List schema；entity 税务状态字段；source 字段形态。

### L4/seam（机制，挂 open boundary）
- Change List Engine 依赖图的声明位置/形态；Finalization 写入机制调用契约/顺序/原子；授权确认机制 exact 实现；确定性发现 job 调用/节奏/阈值；read-back UI 与复核视图是否同一 shell。

### L2·外阻（圈外，挂起不填）
- Intervention Log 正式 spec；Governance Log 新建；确定性发现 / 授权确认机制 / Finalization 机制（另窗细化）；语义发现器 merge/split（挂起，另窗）。
