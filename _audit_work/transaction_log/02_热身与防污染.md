## 1. 行话 / 领域速记（帮主线程快速理解 B 大概在讲什么；非权威）

- **final audit-trail persistence node**：B 把 Transaction Logging 定位成「最终审计轨迹持久化」节点，位于 finalizable outcome / 必要 review / JE generation 之后。`transaction_logging_node__design.md:1`
- **audit-facing `Transaction Log`**：B 说它写入「面向审计的最终交易记录」，只写和查询、不参与 runtime decision、不是学习层。
- **finalization authority / final logging authority**：B 反复用的概念——「当前交易是否够资格被最终记录」；B 把「authorized / pending / review_required / not_approved / governance_blocked / je_blocked / traceability_blocked / conflicting」列为 `final_logging_authority_status` 候选值。`transaction_logging_node__design.md:516`
- **blocked handoff / consistency issue signal**：B 设了「不能正常落账时的安全停止 handoff」和「一致性问题信号」两类 runtime-only 输出。
- **terminal non-JE outcome**：B 反复挂起的开放问题——JE-blocked / not-finalizable 交易是否需要一条单独的 terminal 审计记录。
- **duplicate-final-log prevention**：B 提的「防止同一交易重复落盘」边界（idempotency key / correction flow 留后续）。
- **candidate handoff preservation**：B 说落账时可保留 case memory / entity / rule / governance 等候选「关系」，但只表示「后续应评估」，不写长期 memory。
- 相邻：**JE Generation Node**（B 称纯分录计算节点，借贷平衡 + tax 一致性，不选 COA、不判 HST/GST、不写 Transaction Log）；**Case Memory Update Node**（B 称完成交易的 case-learning 节点，凭 finalization 沉淀 Case Log，不写 Transaction Log）。

## 2. B 触及、值得问用户的候选问题（只列问题，不给答案；这些点 A 桶未冻结，B 把它们摆了出来）

1. JE-blocked / not-finalizable / terminal non-JE outcome 是否需要一条**单独的 terminal audit-facing durable record**？还是只能停在 intervention / review / blocked 语境？（A 接缝里 JE↔Transaction Log handoff 本就 open。）
2. **correction / reversal / superseded transaction / split transaction / reprocessing** 如何影响既有 Transaction Log？（A 桶 Case Log / Entity Log / Alias Log 多处把这条标为「未冻结」。）
3. Transaction Log 与 **final output report / export artifact** 的 exact field-level boundary 在哪？（什么进审计日志、什么进对外报表。）
4. **Review Node 是否覆盖所有 final logging 前交易**，还是只覆盖 review-required / sampled / governance-candidate？（影响 Transaction Log 触发面。）
5. 哪些 **high-confidence system outcome** 可以不经逐笔 review 直接进入 JE / final logging？（牵涉自动化 authority，A 桶 CJ 也未冻结。）
6. **duplicate final logging prevention** 的 idempotency key / correction flow / historical supersession 行为如何定？
7. review / intervention trace 与 Transaction Log 的 **exact persistence boundary**（什么 review 痕迹进 Transaction Log，什么留 Intervention Log）？
8. final logging blocked handoff 应**返回哪个节点**（Coordinator / Review / JE Generation / upstream rework / governance）？
9. Transaction Log 与 Case Memory Update / Governance Review / Knowledge Compilation / Post-Batch Lint 的 **exact downstream handoff contract**？
10. reasoning / audit narrative 进 Transaction Log 的字段形态——B 给了 `audit_narrative` / `authority_trace` / `audit_trace_refs` 等命名，是否与 A 锁的「reasoning 不成为 authority」一致需用户确认。

## 3. 防污染小抄（B 与 A 明显分叉处，逐条标「此处别顺着 B 走，以 A 为准」）

> 提醒：B 给出的所有字段名 / 枚举值 / 对象结构都**不是** A，契约面字段级 schema 在 A 桶里几乎全是 △缺口（L3/L4/seam）。下列是 B 已具体化、但 A 尚未冻结或表述不同之处，主线程切勿把 B 的具体化当成已锁事实。

1. **B 已铺满字段级 schema**（`transaction_log_entry` / `transaction_logging_request` / `final_outcome_context` 等成表的 Required / Optional 字段、`outcome_status` / `authority_path` / `entry_status` 等枚举）。→ **别顺着 B 走，以 A 为准**：A 桶把「reasoning 在 Transaction Log 的 exact 存储字段」「rule-hit ref 字段形态」「多 log finalization 顺序」全标为未冻结 △缺口；B 的字段表只是它自己的草案，不能当契约面输入或结构模板。

2. **B 自定 `entry_status` 只允许 `final_logged`，且把「能写日志」与 finalization authority 显式解耦**。→ 与 A 方向不冲突（A 也说 Transaction Log 是最终审计、不参与 runtime decision），但 **authority 顺序的权威表述以 A 为准**（Case Log 02:121「Transaction Log 或等价 finalization source 对 final outcome 和 audit trail 优先」）；不要引用 B 的枚举值作为已锁结论。

3. **B 把 finalization trigger order 隐含成「JE Generation → Transaction Logging」单线**（定位段、upstream handoff consumed）。→ **别顺着 B 走，以 A 为准**：A 把「多 log finalization 顺序（Evidence / Transaction / Case / Rule / JE）」「Case Log 与 Transaction Log 的 exact trigger order」明确标为 **未冻结 L4/seam**；不得把 B 的单线顺序当已定。

4. **B 让 Transaction Logging Node 自己「保留 candidate relationships」并产出 `post_logging_handoff` 给 Case Memory Update / Governance / Knowledge Compilation / Post-Batch Lint**。→ **别顺着 B 走，以 A 为准**：A 桶里 Case Memory Update Node 等均为**未落文邻居**，这些下游 handoff 在 A 里是 open boundary；B 的 handoff 对象 / 字段不能进契约面。

5. **B 引入「Transaction Logging Node」作为独立写入节点 + `transaction_log_write_receipt`、`final_logging_blocked_handoff`、`logging_consistency_issue_signal` 等专属对象**。→ **别顺着 B 走，以 A 为准**：A 从不命名「谁写 Transaction Log」——CJ / Coordinator / Evidence Intake 一致只说「由后续 finalization / audit / learning 机制」「Transaction Log 或等价 finalization 机制」承接，writer / 写入机制是 △缺口。是否存在一个独立 logging 节点本身属待审计问题，不可因 B 命名了就当存在。

6. **B 出现「Transaction Identity Node」作为 `transaction_id` source of truth**。→ **别顺着 B 走，以 A 为准**：A 桶（已锁事实 / `当前任务状态.md`）记载 **Transaction Identity 节点已删除**，`transaction_id` 铸号并入 **Evidence Intake 流程第 9 步**。B 这里直接与 A 已锁事实分叉，务必以 A 为准。

7. **B 给 `transaction_context` 列了 `transaction_id / transaction_date / amount_abs / direction / bank_account / evidence_refs` 等具体字段且 `direction ∈ {inflow, outflow}`**。→ **别顺着 B 走，以 A 为准**：这些是 B 的字段草案；A 未在 Transaction Log 契约面冻结任何字段，引用需降级为待论证假设。

8. **B 把 `Intervention Log` / Review context 当作 Transaction Logging Node 可直接读取的输入**。→ 与 A 的「交互事实归 Intervention Log、结果审计归 Transaction Log，不互相替代」分界**方向一致但读取关系是 B 自定的**；A 未冻结 Transaction Log 读 Intervention Log 的 contract（Intervention Log 本身未落文）。以 A 的分界语义为准，不引 B 的读取字段。
