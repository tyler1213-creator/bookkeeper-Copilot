# 01_契约面与缺口 — Finalization（只摆 A 原文 + 标 △缺口；B 类零出现）

> Finalization 是「地板」，没有独立 00/01/02 正式文档。它的契约面 = **每个已落文邻居在自己正式文档里、把 writer / finalization 指向「统一写入机制」的那些原句**。下表逐缝抄 A 原文，semantic 对齐留主线程。

## 一、A 桶（已落文邻居）—— 调用方（供 payload 一侧）

### 1. Entity Resolution Node（正式）
- ER 02:51-59「交给记忆 / finalization 层持久化的内容」：ER 产出 stable entity 本体 + 最小创建 provenance；「实际由 ER 亲手写入，还是由 ER 同步调用专门写入/存储节点立即完成，属于 L4/seam」。
- ER 02:148 `candidate_signal` consumer 含 Case Memory Update / Governance Review；「本节点不直接写入长期记忆」。
- ER 02:301（open boundary）「ER 判定新建 stable entity 后的实际写入执行者、调用方式、写入顺序和多 log finalization」未冻结。
- → **接缝**：ER 调 Finalization 的 `create`（建 stable entity，免凭证，需全 schema）。**△缺口**：调用契约 / payload schema 在 ER 侧标 L4，未落到 A。

### 2. Case Judgment Node（正式）
- CJ 02:92「High Confidence … 应由后续 finalization / Transaction Log 机制按其资格保存；有效性来自 CJ runtime judgment、deterministic COA 校验和 finalization 对本次交易完成状态的认定」。
- CJ 02:93-94、98、184：CJ 只产 `case_memory_update_candidate`、Pending handoff；「不授权写入」「必须等交易 finalized 后凭 transaction_log_ref / finalization proof」。
- → **接缝**：CJ 结果经 Finalization `append` 入 Transaction Log（免凭证）。**△缺口**：finalization「对交易完成状态的认定」语义谁定 —— 是 Finalization 还是上游？A 未明。

### 3. Rule Match Node（正式）
- RuleMatch 02:60「RuleMatch 命中的交易应由后续 finalization 路径写入 TransactionLog」；02:63「TransactionLog/CaseLog 的实际写入者、触发顺序、多 log finalization … 属 L4/seam」；01:113「runtime-only，不直接执行 durable write」。
- → **接缝**：同 CJ，经 Finalization `append`，免凭证。

### 4. Coordinator Node（正式）
- Coordinator 02:61「accountant 明确确认 stable entity 时，本节点发出『stable entity 出现』信号，触发与 ER 共用的统一写入节点处理 Entity Log + Alias Log」；02:62 Case Log 学习记录凭 finalization proof；02:64「最终交易结果审计应由 Transaction Log 或等价 finalization 机制承接；exact contract 属圈外」。
- Coordinator 02:71-74「绝不能裸写」Entity/Alias/Case/Rule/Transaction/Intervention/Governance Log；02:102「durable write 是否可执行」非本节点决定。
- → **接缝**：Coordinator 发**信号**触发 Finalization（身份确认免凭证）。**△缺口**：「发信号」vs「供 payload 直调」——与 ER「共用统一写入节点」的 exact 调用形态 A 未冻结（缺口地图 §182）。

### 5. Human Review Node（正式）—— 最富的调用方规范
- HRN 01:27「备好符合各 log Finalization 要求的 input，交 Finalization 落盘；本节点自己不是 durable write 执行者」；01:35「放行权在 Finalization 凭证校验死代码」；01:66「本节点备料，Finalization 执行跨 log 持久化和凭证校验」。
- HRN 02:50-55「交记忆/finalization 层」：Transaction Log correction append、Intervention Log record、Entity/Rule Log 扩张型 mutation input；「authority 来源是会计师 read-back 签字 + Finalization 凭证死代码强制」。
- HRN 02 §5「Finalization 死代码可以决定」:80-83：① 凭证校验 = 放行权，缺凭证拒绝扩张型写入；② 多 log 原子/顺序/幂等/append-only 不变量，exact 机制留 L4/seam。
- HRN 02 §6 输出契约面（对 Finalization 的 5 类 payload）:correction record→Transaction Log(经 Finalization)；先例更新/supersede→Case Log(经 Finalization)；扩张型 mutation 备料→Entity/Rule Log(via Finalization)；扩张型变更审计→Governance Log(待建)；Intervention 记录+ID→Intervention Log(待建)。
- → **接缝**：HRN 供「一串混合动作 + 凭证」（create+update+N×append+supersede）。这是 Finalization **凭证闸门 + 跨 log 原子**的主用例。A 这侧最完整。

## 二、A 桶（已落文邻居）—— 写入目标 log 一侧

每个 memory layer 02 都把 writer/finalization 显式指向「统一写入机制」并标未冻结：

- **Transaction Log** 02:34「统一 finalization 写入机制（exact writer 未冻结）」直 finalization，写入即定稿、无 candidate 层；02:35 correction append 路径；02:37/168「exact writer、finalization trigger order、多 log 统一写入机制均为 L4/seam」；02:77 append-only。
- **Entity Log** 02:39「direct stable entity creation via finalization / memory write mechanism（实际机制未冻结）」，免 governance approval；02:113「accountant explicit identity confirmation 后由哪个 finalization/memory write mechanism 执行」未冻结；02:116 merge/split 跨 log finalization 未冻结。
- **Alias Log** 02:40「待 L4/seam 冻结的 memory write / finalization 执行机制」，够格即自动落库、免额外 approval；02:105 倾向与创建 stable entity 同时写入但待 seam 确认。
- **Case Log** 02:32「Case Memory Update Node / unified finalization mechanism（候选写入机制，未冻结）」；02:68 invalid/superseded「由 correction/finalization/governance result 触发，机制未冻结」。
- **Rule Log** 02:64「写入执行者和写入顺序留 Governance/L4 seam」；02:40-41 approved/governance mutation 需 accountant 批准（= 扩张型，要凭证）；02:61 candidate 经 sign-off 才进。

→ **接缝结论**：所有 5 个 log 的 durable writer **A 侧均明文留空、统一指向本机制**。这是 Finalization 存在性的最强 A 类支撑，也意味着**本机制是这些 open boundary 的统一收口点**。

## 三、△缺口清单（A 锁不住 / 邻居未落文 → 交用户判断，本轮最值钱产物）

G1. **动作集是否恰为 create/update/append/supersede 四个**，及各动作 payload exact schema、动作类型 enum（B §四/§九 标 L3）。A 各 log 只给「写入类型」(direct finalization / correction append / approved-governance mutation / candidate)，未给统一动作枚举。→ 对齐问题。
G2. **凭证 exact 形态** + 凭证如何从「会计师签字」生成、如何随 payload 传递（B §五标 L3；缺口地图 §255 L3）。
G3. **多 log finalization 的写入顺序 / 原子 / 回滚 / 幂等键**（缺口地图 §43、§224、§262 全标 L4/seam）。→ 实现级，是否本轮只锁契约、不锁机制？
G4. **「单一全局写手」vs「共享库 + 单一事务边界」**（B §三/§九 标 L4，明令别现在锁死）。
G5. **Governance Log 耦合契约**:每笔扩张型变更落盘时同步写 Governance Log 一条审计 —— 这条「耦合时序」算 Finalization 的硬契约还是 Governance Log 自己的？Governance Log **未落文/待建**（缺口地图 §19、§271）。→ 挂 open boundary，**绝不许拿 B 填**。
G6. **「finalization 对交易完成状态的认定」语义归属**:多个节点 02 说「有效性来自 finalization 对本次交易完成状态的认定」(rule_match 02:60、CJ 02:92)。这是否意味着 Finalization 做了超出「怎么写」的判断（与 B §七「不重判该不该写」张力）？→ 需用户厘清边界。
G7. **免凭证写入的「写入即定稿 vs 候选」边界**:Transaction Log「写入即 finalization 定稿、无 candidate 层」(02:43)，但确定性发现 job「投候选进 inbox」也走这套库 —— 同一地板既定稿又投候选，candidate 语义是否属 Finalization？（审核 inbox 未落文，缺口地图 §272）。
G8. **失败回滚的善后边界**:B §六说地板只回滚 + 如实上报、善后归调用方。A 侧 Evidence Log「finalization、失败回滚」标 L4(缺口地图 §41)，未确认调用方契约。
