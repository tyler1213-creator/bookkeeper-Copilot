# 审计目标与原则

## 文件职责

`审计目标与原则.md` 是这个审计仓库的稳定宪章。

一句话职责：说明这个审计为什么做、用什么长期原则判断设计，不记录当前进度或具体设计结论。

它保存长期稳定的项目目的、产品原则和审计约束。

它不负责：

- 当前任务交接
- 临时工作重点
- 当前风险
- 下一步行动
- 审计阶段执行细节
- 具体主题的已确认结论记录
- node / field / memory layer 的具体结论

操作规则看 `AGENTS.md`，实时交接看 `当前任务状态.md`，审计规划看 `审计阶段路线图.md`。具体主题的已确认结论应回写到用户指定的长期结论文档、对应主题记录或正式 `BK_Copilot/` 文档，不再维护单独决策文档。

## 项目目的

这个仓库用于从第一性原理审计当前 AI bookkeeping 系统设计。

审计目的不是维护 Hermes，不是重建旧系统，也不是证明新系统天然更好。它的任务是站在真实产品目标上，判断这个系统到底应该长什么样。

## 核心产品目标

系统应该：

> 帮助会计师把客户历史账务和过往判断沉淀成可复用记忆，让新交易自动获得有证据支持的记账建议，并通过会计师纠正持续学习，在不牺牲审计性和控制权的前提下逐步提高自动化率。

这个目标是长期稳定的审计标准。

凡是不能清楚支持记忆复用、有证据支持的记账建议、基于纠正的学习、审计性、会计师控制权或自动化率提升的设计，都应该被质疑。

## 稳定产品语境

产品领域是面向会计工作流的 AI 辅助记账，包括：

- 客户历史账务记录
- 会计师判断和纠正
- 有证据支持的分类建议
- journal entry 生成
- 审计追溯性
- 长期记忆和治理

## 仓库边界

这个仓库当前的有效设计来源以现行 BKCopilot 内容为准，尤其是：

- `BK_Copilot/` 下已落文的正式 node / memory layer / shared mechanism 草案
- active 治理 / 跟踪文档
- 用户指定的主题记录或已审提案

仓库中仍保留两组历史材料，但它们不属于默认审计目标：

- `new system/` 下 New System 材料已经作废，不再作为审计目标。
- `old_system_nodedesign/` 下 Old-System 材料只保留为历史材料，不作为审计目标、历史意图权威、标准答案或回归目标。

未经用户明确同意，不得读取 New System 或 Old-System 文档内容。即使用户授权读取，它们也只能作为历史 source text / 追溯背景；不能证明某个设计正确、必要、应保留、应回归或应作为 baseline。如果其中某个想法看似有用，必须脱离来源重新按核心产品目标、当前已确认规则和正式 `BK_Copilot/` 草案论证。

## 稳定审计原则

- 第一性原理优先于对旧架构或当前架构的忠诚。
- 除非复杂度保留了真实且必要的能力，否则更简单的结构优先。
- 每个 node、field、log、memory layer 和 workflow step 都必须证明自己为什么需要存在。
- 一个组件不应仅仅因为架构看起来更完整而存在。
- 如果删除、合并或内联某个对象不会实质损害核心产品目标，那么它大概率不具备独立存在的充分理由。
- 审计性和会计师控制权是核心能力，不是附加装饰。
- 自动化只有在有证据、已批准记忆、适当 authority 和可恢复 audit trail 支持时才有价值。
- 从纠正中学习必须改善未来行为，但不能让一次性错误污染 durable memory。
- 节点与记忆层之间只通过显式、最小、写明的契约（接口）交互，而不依赖彼此的内部实现或隐式假设；这是降低有害耦合的核心手段。流水线式的顺序依赖不可避免也不必规避，真正要保持松的是接口面，而不是顺序本身。这条原则不要求每个对象一产出就给出冻结的字段级 schema——契约的最终冻结属于 data contract 阶段（workflow Stage 3 / memory M3）；更早的阶段只需明确意识到这一点并朝它设计。
- 任何历史材料或待审对象中的设计想法都不能因来源获得 authority；必须按当前产品目标重新论证其必要性。

## 历史材料处理约束

未经用户授权，不读取 New System 或 Old-System 文档。

在用户授权读取后，从 New System 或 Old-System 中看到任何可取之处时，不要把它当作可继承约束，而要把它降级为待论证假设。

正确问题不是“某个设计材料里有哪些 node / field / flow 存在过”，而是：

- 这个设计声称要满足什么用户需要？
- 这个需要现在是否真实存在且重要？
- 是否有更简单、更可审计、更可控的方式满足它？
- 如果要保留，究竟是哪一个当前产品约束支撑它？
